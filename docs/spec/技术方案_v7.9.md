# 错题管理系统 · 技术方案文档 v7.9

> **这个系统干一件事：帮你在考试前拿到尽可能多的分。**
>
> 得分 = **存量**（稳固了的，不靠冲刺也能答对）+ **增量**（考前高强度复习，短暂激活进入的临时状态）
>
> **系统负责两件事：**
> - **建存量**——错过的题，AI诊断思维链，间隔复习反复巩固，直到掌握度≥80%且不反弹，锁定为压箱底实力
> - **备增量**——临考前自动识别"差一口气就能激活"的考点，集中推给用户快速过一遍，不引入新知识只做激活

---

## 🎯 设计初衷

这个系统是为以下三类人打造的：

1. **想要学习** — 有主动意愿，需要系统把意愿转化为有效行动，而不是靠自律硬撑
2. **每天至少刷题 70 道** — 默认总量70道，错题到期优先全推，真题补足剩余名额，用户可自行调整
3. **能坚持、想要高分** — 目标明确，需要持续的正反馈和留存激励，让每一天的坚持都有看得见的回报

---

## 👤 用户的一天

> 用户感知到的全部交互，就这四步：

1. **打开 App** → 看到今日任务（目标总量默认70道，用户可设置）：
   - 📕 **错题复盘**：今日到期错题优先全推，超过上限按 errorROI 截断顺延
   - 📄 **真题刷题**：剩余名额自动补满，按考点频率+未做过优先排序
2. **一道一道做** → 选答案 → 揭晓对错 → 看解析（错题可选深度or快速）
3. **做完** → 看今天涨了多少存量分
4. **关掉**

重点是质量——错题到期必须过，真题保量。其余所有功能——遗忘曲线、errorROI排序、AI诊断——全在后台默默运行，用户无需感知。

> ⚠️ **考前≤14天**：错题复盘自动替换为增量候选队列（mastery 60-80%），快速过不引入新知识；真题改为按题型专项刷，总量自动降为50道。

## 〇、核心逻辑：两阶段分治

> 本系统的所有设计决策，都服务于同一件事：**最大化得分 = 存量（稳固不反弹）+ 增量（临考激活的临时状态）**
>
> 两阶段不同目标，不同策略：**建设期（考前60天→14天）深度存量化 / 激活期（临考≤14天）快速激活增量**

### 0.1 存量化的本质是资源分配问题

用户每天只有有限的练习时间。存量能力增速 = f(练哪些题) × f(怎么练)。

- **怎么练** → 已解决：修正思维链（先选答案→写思路→揭晓），AI验证真实思维，间隔复习巩固
- **练哪些题** → 不是"缺口最大的"，而是**"今天最有可能推过存量化门槛的"**

> **存量化门槛**：masteryPercent ≥ 80% 且最近2次复习均未下降超过10% 且最近1次非isSlowCorrect
> → 达标的考点视为"稳固资产"，计入存量分；反弹时自动降级

### 0.2 错题的存量化价值不等

```
同样花10分钟：

  选项A：练 翻译推理（masteryPercent=70%，再练一次大概率推过80%存量化门槛）
          存量化价值：今天很可能锁定这个考点的 0.6分

  选项B：练 数量关系（masteryPercent=10%，距存量化门槛还要5-8次复习）
          存量化价值：今天贡献甚微，且距考试30天内性价比极低

  → 系统应该始终推 选项A
  → 核心逻辑：优先把"快到手的"拿到手，而非总追"缺口最大的"
```

### 0.3 单题存量化价值（核心算法，贯穿全系统）

```
errorROI(每道错题今天练习的存量化价值) =
    stockifyProb × stockifyValue × timeViability

其中：
  stockifyProb = 存量化概率（今天练了，能推过存量化门槛的可能性）
    masteryPercent 60-80%: 0.8   // 差一口气，今日复习大概率推过门槛
    masteryPercent 30-60%: 0.4   // 还需多次，今日可积累进度
    masteryPercent  0-30%: 0.1   // 距门槛太远，今日贡献有限
    修正：overdue=true → ×1.3（已到复习窗口，当日练习性价比更高）
    修正：isHot=true   → ×0.7（连续答错说明思路未通，先让AI重新诊断）

  stockifyValue = 一旦存量化，能贡献多少稳固分
    = scoreWeight[题型] × skillFrequencyWeight
    // scoreWeight：该题型满分，来自 getWeights(examType)（见 §十八.2）
    // skillFrequencyWeight：该考点历年真题出现率加权（见 §二十一.2）

  timeViability = 时间可行性（距考试越近，低掌握度考点越不值得今天深度投入）
    daysToExam > 60天:  1.0（全部考点都值得深度投入）
    daysToExam 30-60天: masteryPercent > 30%? 1.0 : 0.2
    daysToExam 14-30天: masteryPercent > 50%? 1.0 : 0.1
    daysToExam ≤ 14天:  masteryPercent > 70%? 1.0 : 0.0
    // ⚠️ ≤14天触发激活期切换：建设期队列（深度诊断+AI验证）自动让位给 §0.5 增量激活队列
    // 此时 mastery < 70% 的题不是"放弃"，而是转入"快速过一遍"模式，目标是激活而非存量化
```

**该公式驱动的系统行为：**
- 今日任务（§16.3）：优先推"最接近存量化门槛"的考点，而非"缺口最大的"
- 练习结束反馈（§16.6）：展示"存量化进度 +X%"，而非"+X分"
- 战略建议（§16.8）：daysToExam ≤ 30 且 masteryPercent < 30% 的考点 → 建议战略跳过
- 冷启动推荐（§二十一.4）：新用户无错题时，用 ExamTopicStats.frequency × difficulty 生成首推队列

### 0.4 分值体系与存量化进度（全文唯一标准）

**行测满分 ≈ 100分**（各考试体系统一约定，具体权重见 §十八.2）

- 国考参考：言语32 + 判断36 + 资料20 + 数量15 + 常识12 ≈ 115（折算为100后各比例不变）
- §十六.1 旧公式（满分≈150）**作废**，统一以 `getWeights(examType)` 为准

**用户看到的核心指标：存量底线分 + 增量空间分（两条线，一眼看懂缺口在哪）**

```
存量分 = Σ(已存量化的考点 × scoreWeight × skillFrequencyWeight)
  其中"已存量化"= masteryPercent ≥ 80% AND 最近2次复习未下降超过10%

增量空间分 = Σ(增量候选项的考点 × scoreWeight × skillFrequencyWeight)
  其中"增量候选项"= masteryPercent 60-80%（见 §0.5，差一口气、适合考前激活）
  // ⚠️ 上限80%（与§0.6状态机保持一致，避免75-80%区间遗漏）

存量化进度% = 存量分 / targetScore × 100

示例（国考，targetScore=85）：
  存量底线：56分（已稳固，不冲刺也能有）
  增量空间：≈12分（mastery 60-80% 的候选项，考前集中激活可拿）
  当前预期：56 + 12 = 68分  目标：85分  缺口：17分

  展示：
  存量  ██████████████░░░░░  56分 · 已稳固
  增量  ████░░░░░░░░░░░░░░░  12分 · 可激活
  缺口  → 17分 = 存量不足（mastery < 60% 的考点）还是增量候选太少？
```

> **存量分**的含义：不靠任何临考冲刺，这些分稳拿。
> **增量空间**的含义：考前几天集中冲刺，合理预期能再激活这些分。
> 缺口来源决定下一步策略：存量不足→继续深度巩固；增量候选太少→优先把mastery推到60%以上。

### 0.5 激活期：增量候选项识别与快速激活（临考≤14天）

建设期和激活期是两套完全不同的练习范式，不能混用。

**增量候选项的识别标准（同时满足）：**
```
① masteryPercent 60-80%   → 差一口气或已接近存量化但稳定性条件未满足，
                             考前2-3天高频复习可推过门槛或维持激活状态
  // ⚠️ 上限从75%改为80%：避免75-80%区间题被推题逻辑遗漏（见§0.6状态机说明）
② 遗忘曲线平缓            → 上次复习后mastery下降 < 5%/天，不会一睡又忘
③ 偏识别类题型            → 言语、判断推理（认出来就对）
                             ≠ 数量关系（算不出来就是不会，冲刺ROI低）
```

**激活期队列生成（daysToExam ≤ 14时自动触发）：**
```typescript
// Prisma ORM 实现（Schema 中题型字段为 question.type，无独立 skillType 字段）
const candidates = await prisma.userError.findMany({
  where: {
    userId,
    masteryPercent: { gte: 60, lte: 80 },  // ⚠️ 上限80（非75），见§0.6状态机修正说明
    decayRatePerDay: { lt: 0.05 },
    question: {
      type: { in: ['判断推理', '言语理解', '资料分析'] }
      // ≠ 数量关系：算不出来就不会，冲刺 ROI 低
    },
  },
  orderBy: [
    // 最接近激活门槛（75%）的优先——差一口气更值得推
    { masteryPercent: 'desc' },
    // 同等 mastery 下，历年真题出现率高的考点优先
    // （skillFrequencyWeight 需 JOIN ExamTopicStats 计算，见 §二十一.2）
  ],
  take: 30,  // 集中火力在最接近可用状态的 30 个考点
  include: { question: true },
})
```

**激活模式 vs 建设模式（两套练习范式）：**

| | 建设期（存量化）| 激活期（增量激活）|
|--|--|--|
| 目标 | 推过存量化门槛，永久稳固 | 临时激活到可用状态 |
| 练法 | 先选答案→写思路→揭晓→AI验思维链 | 见题→给答案→继续，不停下来深度诊断 |
| 时间 | 每题5-10分钟 | 每题1-2分钟，高频过 |
| AI介入 | 每题诊断，找根因 | 仅错误率>50%时介入，其他快速过 |
| 新知识 | 可以引入 | **绝不引入**，只激活已有记忆 |

**激活期系统行为：**
- 今日任务（§16.3）：展示"增量激活清单"，而非存量化队列
- 练习结束反馈（§16.6）：展示"已激活 X 个考点，预计可多拿 Y 分"
- 战略提示：`daysToExam ≤ 7` 时，提示"现在只做激活，不引入新题"

### 0.6 题目四状态机（全系统统一语言）

每道错题在任意时刻必处于且仅处于以下四种状态之一，驱动推题逻辑和 UI 展示：

```
┌──────────────┬────────────────────────────────────────────────────────┬──────────────────────────────┬──────────────────┐

├──────────────┼────────────────────────────────────────────────────────┼──────────────────────────────┼──────────────────┤

│ 🔥 增量候选  │ mastery 60-80%（含75-80%过渡区）                        │ 激活期首推，建设期次推         │ 临考冲刺目标     │
│ 🔨 建设中    │ mastery 30-60%                                          │ 建设期主力，errorROI驱动       │ 攻坚中           │
│ ⏸ 本次放弃  │ mastery<30% AND daysToExam<30                           │ 不推送，战略跳过提示           │ 本次跳过         │
└──────────────┴────────────────────────────────────────────────────────┴──────────────────────────────┴──────────────────┘

// ⚠️ 增量候选范围调整说明：
// 原设计 mastery 60-75% 存在 75-80% 空白区（不属于任何状态，推题逻辑会忽略该区间题目）。
// 修正为 60-80%：mastery 已很高但尚未满足稳定性条件的题，优先激活价值高，应归入增量候选。
// 激活期 §0.5 的 Prisma 查询需同步改为 { masteryPercent: { gte: 60, lte: 80 } }。

状态转移：
  建设中 → 增量候选：mastery 升到 60%
  增量候选 → 存量化：mastery≥80% AND 近2次未降>10% AND 最近1次非isSlowCorrect
  存量化 → 增量候选：反弹，mastery 下降超过 10%（reboundAlert=true）
  建设中 → 本次放弃：daysToExam < 30 AND mastery 仍 < 30%（系统提示，用户可忽略）
  本次放弃 → 建设中：用户手动取消跳过
```

### 0.7 五个增益手段（补充"道"的达成路径）

> 以下 5 个手段是对核心框架（存量+增量）的增益，不改变架构，只需在现有功能基础上叠加。

---

#### ① 考前48小时超级激活（优先级：最高，成本最低）

**问题**：≤14天激活期是通用节奏，最后48小时有独特的生理窗口——睡前复习在REM睡眠中巩固，是记忆效率最高的时刻。

**策略**：

```
daysToExam ≤ 2 时，触发"超级激活清单"（替代常规今日任务）：

  1. 把所有 isStockified=true 的考点快速过一遍（防临考遗忘，每题30秒）
  2. 增量候选清单（mastery 60-80%）再刷一次（强化短期记忆进入激活状态）
  3. 绝不出现 mastery < 60% 的题（此时引入不会的只会增加焦虑）
  4. UI提示："今天不学新知识，只激活已有记忆"

预期效果：相比普通复习，最后48h这套策略可额外激活1-3分
```

**实现**：触发条件 `daysToExam ≤ 2` 时，今日任务接口改为超级激活清单查询。

---

#### ② 计时训练模块（优先级：高，数据已有）

**问题**：isSlowCorrect 已标记"慢正确"，但没有针对速度的训练。mastery=80% 但每次慢正确 ≠ 真存量化（实考会因超时失分）。

**触发条件与规则**：

```
当某道题连续 3 次 isSlowCorrect=true 时：
  → 标记为"需提速"，进入计时训练队列

计时训练模式：
  · 答题时间限制 = 该题型警戒线 × 60%（比正常限制更严）
  · 在限制时间内答对 → 清除 isSlowCorrect 标记，mastery 不惩罚
  · 超时仍答对 → 维持 isSlowCorrect，继续计时训练
  · 答错 → 正常降级

isStockified 判断规则补充：
  masteryPercent ≥ 80% AND 近2次未降>10% AND 最近1次非isSlowCorrect
  // 慢正确无法真正存量化
```

#### ③ 错误陷阱聚合识别（优先级：中，需数据积累）

**问题**：逐题 errorReason 只分析单道题，无法发现用户的系统性认知偏差模式。

**实现**：

```typescript
// 每周定时任务：分析用户最近30道错题的 errorReason 分布
async function analyzeErrorPatterns(userId: string) {
  // ⚠️ Schema 字段名为 isCorrect（非 answerCorrect）
  const recentErrors = await prisma.reviewRecord.findMany({
    where: { userId, isCorrect: false },
    orderBy: { createdAt: 'desc' },
    take: 30,
    include: { userError: { include: { question: true } } },
  })

  // 按 errorReason + 题型 聚合
  const patternMap = recentErrors.reduce((acc, r) => {
    const key = `${r.userError.question.type}::${r.userError.errorReason}`
    acc[key] = (acc[key] ?? 0) + 1
    return acc
  }, {} as Record<string, number>)

  // 出现 ≥ 3 次的模式 → 生成专项训练提示
  const hotPatterns = Object.entries(patternMap)
    .filter(([, count]) => count >= 3)
    .sort(([, a], [, b]) => b - a)

  return hotPatterns
  // 示例输出：[["言语理解::过度推断", 5], ["逻辑判断::混淆充分必要", 4]]
}
```

**UI 展示**（周报卡片）：
```
📊 你的错误模式（本周）：
  言语理解：5次 被"过度推断"选项迷惑  → 今日推3道专项训练
  逻辑判断：4次 混淆充分必要条件        → 建议重看规律
```

#### ④ 考场答题顺序策略（优先级：中，纯输出不需要额外数据）

**问题**：系统只管练习，不管考场如何分配时间。mastery 高的题型先做 → 稳拿分，减少焦虑；mastery 低的放到最后（或战略放弃）。

**生成逻辑**：

```typescript
// 根据各题型的 isStockified 比例，生成考场答题顺序建议
function generateExamStrategy(userSectionStats: UserSectionStats[]) {
  return userSectionStats
    .map(s => ({
      section: s.section,
      stockifiedRate: s.stockifiedCount / s.totalCount,
      suggestion:
        s.stockifiedRate >= 0.7 ? '先做·稳拿分' :
        s.stockifiedRate >= 0.4 ? '正常顺序' :
        daysToExam <= 30        ? '最后做·能做多少做多少' :
                                  '继续攻坚'
    }))
    .sort((a, b) => b.stockifiedRate - a.stockifiedRate)
}
```

#### ⑤ 记忆锚点生成（优先级：长期，AI调用1次永久复用）

**问题**：题目存量化后进入30天守卫队列，但用户在考场提取记忆时，不会想起"这道题我做对过"，而是需要一个**关键词触发器**。

**实现**：

```typescript
// 题目首次达到 isStockified=true 时，为当前用户生成个人记忆锚点（仅调用1次）
// 优先使用 question.sharedMemoryAnchor（共享，节省AI费用）；
// 若题目无共享锚点，则为本用户单独生成并存入 personalMemoryAnchor
async function generateMemoryAnchor(userError: UserError & { question: Question }) {
  if (userError.personalMemoryAnchor) return  // 已生成，不重复
  if (userError.question.sharedMemoryAnchor) return  // 共享锚点已存在，直接复用

  const prompt = `
    题目：${userError.question.content}
    正确解法：${userError.question.sharedAiAnalysis ?? userError.customAiAnalysis}

    请用1句话（≤20字）总结这道题的解题关键词/口诀，
    让考生在考场看到类似题时能立刻想起解法。
    格式：仅输出这1句话，不要解释。
  `
  const anchor = await callMinimax(prompt)
  await prisma.userError.update({
    where: { id: userError.id },
    data: { personalMemoryAnchor: anchor },
  })
}

// 前端展示时取锚点的优先级：
// personalMemoryAnchor（用户个性化）> question.sharedMemoryAnchor（共享默认）
const displayAnchor = userError.personalMemoryAnchor ?? userError.question.sharedMemoryAnchor
```

**Schema 字段说明**（双层架构，见§三）：
```prisma

// 用户个人AI生成的1句话记忆锚点；未设置时 fallback 到 question.sharedMemoryAnchor

// Question 层（公共共享，由管理员触发或首个触发者生成后沉淀）
sharedMemoryAnchor String?
// 题目级别的共享锚点，所有用户默认复用，节省90%+ AI费用
```

**考前使用场景**：
```

  只展示已存量化考点的锚点（个人锚点 > 共享锚点，不出题）
  用户滑动确认：知道了 / 再想想
  5秒一条，20分钟过完所有已存量化考点
  → 相比重做题，效率提升10倍，且不引入焦虑
```

### 0.8 冷启动补救机制：真题上传 → 自动生成练习队列

> **解决问题**：新用户没有错题 / 错题极少，推题逻辑无数据可推，Day 1-7 体验近乎空白。

#### 核心流程

```
用户上传一批真题（PDF / 截图 / 手动批量）
        ↓
AI 解析 → 写入 Question 表（按考点分类，isPublic=true）
        ↓
        ├── 有错题记录（UserError）
        │     → 错题复习优先（errorROI排序）
        │     → 复习间隔期 → 真题练习补位
        │
        └── 无/少量错题（冷启动模式）
              → 从上传真题中按 ExamTopicStats.frequency 排序推送
              → 做对 → 记录"已正确"，7天内不重推，按题型统计答对率纳入预测分
              → 做错 → 自动创建 UserError → 触发正常复习流程
```

#### 两种模式的任务调度

| 状态 | 今日任务来源 | 优先级 |
|---|---|---|
| 有到期复习任务 | UserError 复习队列（errorROI驱动）| 最高 |
| 复习做完 / 间隔期 | 上传真题练习（frequency驱动）| 次之 |
| 无任何数据 | 系统提示上传真题（引导入口）| 兜底 |

**核心原则：不允许出现"今天没有任何任务"的空白日。**

#### 真题用完三级降级策略

当上传的真题全部做完（未做过的题 < 今日名额）时，按以下顺序补量：

| 级别 | 来源 | 触发条件 |
|-----|------|---------|
| 1级 | AI变式题 | 优先生成，同考点换领域/换难度 |
| 2级 | 30天前旧题循环 | 做对超30天的题重新推送，考察长期记忆 |
| 3级 | 提示上传更多真题 | 以上均不足时，首页引导入口 + 推算"需上传X套" |

#### 做错自动入错题本

```typescript
// 用户在真题练习中答错 → 无缝创建 UserError
async function onPracticeWrong(userId: string, questionId: string, record: ReviewRecord) {
  const existing = await prisma.userError.findFirst({ where: { userId, questionId } })
  if (existing) return  // 已在错题本，走正常复习流程即可

  await prisma.userError.create({
    data: {
      userId,
      questionId,
      masteryPercent: 0,
      reviewInterval: 1,
      nextReviewAt: addDays(new Date(), 1),
      errorReason: record.errorReason ?? '',
      // 复用题目级别的共享AI解析，零成本
    }
  })

  // 清除该题的 PracticeRecord（如果之前答过对，现在答错需重置）
  await prisma.practiceRecord.deleteMany({ where: { userId, questionId } })
}
```

#### 做对记录（冷启动期）

> **为什么需要这个函数**：§0.8 承诺"做对 → 7天内不重推 + 答对率纳入预测分"，但做对的题不会创建 UserError，必须有独立存储路径。

```typescript
// 用户在真题练习中答对 → 写入 PracticeRecord（不创建 UserError）
async function onPracticeRight(userId: string, questionId: string, question: Question) {
  // upsert：同一题多次答对时更新 nextShowAt（延后下次推送）
  await prisma.practiceRecord.upsert({
    where: { userId_questionId: { userId, questionId } },
    update: {
      isCorrect: true,
      nextShowAt: addDays(new Date(), 7),  // 7天内不重推（见§0.8核心流程）
    },
    create: {
      userId,
      questionId,
      isCorrect: true,
      nextShowAt: addDays(new Date(), 7),
      questionType: question.type,         // 缓存题型，供预测分聚合
    },
  })
}

// 冷启动推题时过滤已做过（包括答对的）：
const coldStartQueue = await prisma.question.findMany({
  where: {
    isPublic: true,
    NOT: {
      OR: [
        { userErrors: { some: { userId } } },           // 已在错题本
        { practiceRecords: { some: {                    // 7天内答对过
            userId, nextShowAt: { gt: new Date() }
        }}},
      ]
    },
  },
  // ... orderBy / take 同原有逻辑
})

// 预测分聚合时纳入 PracticeRecord.isCorrect（按题型）：
// SELECT questionType, COUNT(*) FILTER (WHERE isCorrect) as correct, COUNT(*) as total
// FROM practice_records WHERE userId = $1
// GROUP BY questionType
// → 与 UserError 的复习答对率合并计算预测分
```

### 0.9 用户留存节点分析（20天模拟 · 产品设计依据）

> 模拟角色：小张，国考备考，距考试60天，每晚40分钟

#### 三个阶段的体验曲线

```
第1-7天（冷启动期） ：期待高 → 数据积累 → Day 7 正反馈真空危机 ← 最高流失风险
第8-15天（习惯期）  ：Day 12 第一个🔥增量候选出现 → 留存用户开始确立
第16-20天（积累期） ：任务量峰值堆积 → 疲惫感 → 但系统价值开始显现
```

#### Day-by-Day 关键节点

| 天 | 关键事件 | mastery变化 | 用户感受 |
|---|---|---|---|
| 1 | 上传真题 + 初始错题录入 | 0% | 期待 |
| 2 | interval=1复习，答对→25%，答错→10% | 建设中 | 有点用 |
| **4** | **空任务日（无§0.8时）** | 无变化 | **困惑，流失风险** |
| **7** | 存量分=0，增量分=0 | 30-45% | **💥"用了一周什么都没有"** |
| **12** | 第一批题mastery=65%，变为🔥增量候选 | 增量候选 | 第一个正反馈 |
| **16** | 任务量峰值（前两周录的题集中到期）| 多条线并行 | 疲惫感 |
| **27** | 第一道题达到✅存量化 | 80%+ 稳定 | 强认可，长期留存确立 |

#### 6个设计漏洞与优化方案

**P0 — 第一道存量化要27天才到达（最快路径）**

```
最快路径：录题(Day1) → interval=1 → 3 → 7 → 15 → 存量化(Day27)

优化A（推荐）：引入"预存量化"里程碑
  mastery ≥ 60% 且连续2次未下降 → 标记为"预稳固 🌱"
  计入增量空间分展示，给用户在Day 12附近的正反馈
  真正存量化仍需 mastery ≥ 80% 且稳定

优化B：前4次全部"答对+思路正确"，允许压缩间隔
  1 → 2 → 5 → 12（而非 1 → 3 → 7 → 15）
  优秀学习者可提前10天看到存量化
```

**P0 — Day 4 空任务日导致流失**
```
解决：§0.8 真题上传补位（已设计）
     间隔期 → 推送真题练习（高频考点，由易到难）
     同时展示倒计时："下次复习在X天后"
```

**P1 — 第7天"0分危机"**
```
解决：Day 7 强制展示里程碑卡片
  ┌──────────────────────────────────────────┐
  │ 本周你完成了 {weeklyReviewCount} 道复习   │
  │ 已有 {progressedCount} 道题掌握度提升     │
  │ 最接近进阶：翻译推理 · mastery 45%        │
  │ 再复习2次即可进入增量候选                  │
  │ 预计第一道存量化：12天后 · 继续保持！      │
  └──────────────────────────────────────────┘
  // weeklyReviewCount = 本周完成的所有复习题数（含错题复习+真题练习）
  // progressedCount   = 本周 masteryPercent 有提升的错题数量
```

**P1 — 第16天任务量峰值堆积**
```
解决：设置错题单日上限（默认30道，用户可调）
  溢出的按 errorROI 降序顺延到次日
  超出时提示："今天任务已满，已安排X道到明天"
  // ⚠️ 见功能清单§63：错题单日上限（默认30）；总目标默认70（含真题补位）
```

**P2 — 建设期"写思路"对新用户门槛太高**
```
解决：前5天进入"轻量入门期"
  允许跳过写思路，AI自动生成思路供参考
  第6天起引导写思路，逐步建立习惯
```

**P2 — 间隔期空白感**
```
解决：§0.8 真题补位 + 每完成5道展示小徽章
  "今日额外练习 +5题 · 超出计划20%"
  积累感替代空白感
```

#### 留存关键时间线（开发优先级参考）


## 二.X、AI 调用预算模型（独立开发必读）

> 成本明细给出了总估算，本节把每个调用点拆开，避免上线后费用失控。

### 各调用点分类

| 调用点 | 触发频率 | 是否可缓存 | 控制策略 |
|--------|---------|-----------|---------|
| sharedAiAnalysis（公共解析）| 每道题仅触发1次（全局）| ✅ 永久复用 | 答题前先查是否存在，存在则跳过 |
| 思路验证（thinkingVerdict）| 深度模式每题1次/用户 | ✅ 缓存到 ReviewRecord | 同一道题同一用户只调用1次，重复复习读缓存 |
| AI首次诊断（aiRootReason等）| 每用户每道新错题1次 | ✅ 存入 UserError | 已诊断过的题不重复调用 |
| customAiAnalysis（个性化诊断）| 用户主动触发 | ✅ 存入 UserError | 用户控制，低频 |
| AI变式出题 | 用户主动 or 规律自测后 | ✅ 存入 AiGeneratedQuestion | 每个 skillTag 设预生成上限（见下） |
| 记忆锚点生成 | 首次存量化时1次/用户 | ✅ personalMemoryAnchor | 已生成不重复调用 |
| 错误陷阱聚合 | 每用户每周1次（≥30条数据后启用）| ✅ 结果缓存7天 | 低频，可接受 |
| 社区洞察蒸馏 | 管理员手动触发 | ✅ 发布后永久复用 | 人工控制，极低频 |

### 高频调用点：思路验证

```
估算（以单用户为例）：
  每天深度模式使用率 ≈ 40%（不是每道题都深度模式）
  每天深度模式题数  ≈ 70 × 40% = 28 道
  每月思路验证调用  ≈ 28 × 30 = 840 次

控制：
  ① 同一道题同一用户：ReviewRecord 中已有 thinkingVerdict → 直接展示，不重新调用
  ② 快速复习模式下，thinkingInputType=null → 不触发思路验证
  ③ masteryPercent > 70% 时，仅30%概率要求写思路（§6.4），进一步降低调用频率
```

### AI变式出题上限

```typescript
// 每个 skillTag 预生成上限：防止无限生成堆积成本
const MAX_AI_QUESTIONS_PER_SKILL_TAG = 20  // 每个考点最多预生成20道变式题

async function canGenerateVariant(skillTag: string): Promise<boolean> {
  const existing = await prisma.aiGeneratedQuestion.count({
    where: { skillTag, isActive: true }
  })
  return existing < MAX_AI_QUESTIONS_PER_SKILL_TAG
}

// 生成前检查上限；超出时优先复用已有变式题（随机选未做过的）
// 用户做完所有变式题后，提示："已练完所有变式题，等系统补充新题"
```

### 月度费用估算（单用户）

```
场景假设：1个活跃用户，每天70题，深度模式使用率40%

首月（冷启动，大量新错题）：
  新错题AI诊断  ≈ 200道 × ¥0.005 = ¥1.0
  思路验证     ≈ 840次 × ¥0.003 = ¥2.5
  变式出题     ≈ 50道  × ¥0.000024 ≈ ¥0.001（可忽略）
  首月合计     ≈ ¥3.5

稳定月（错题库建立后，大量命中缓存）：
  新错题AI诊断  ≈ 50道  × ¥0.005 = ¥0.25（大部分复习旧题，命中sharedAiAnalysis）
  思路验证     ≈ 840次 × ¥0.003 = ¥2.5（仍是主要成本）
  稳定月合计   ≈ ¥2.8

结论：¥3-5/月 的估算合理，思路验证是最大单项成本
```

> ⚠️ **多用户场景**：N个用户时，sharedAiAnalysis 共享带来边际成本趋近于0的效果——第1个用户触发生成，后续N-1个用户零成本。主要增量是思路验证（每用户独立）≈ N × ¥2.5/月。

## 二.6、账号控制（管理员建账号 + 时效密码）

> 你是唯一管理员，控制谁能登录，密码有有效期，到期自动失效。

### 账号表核心字段

```prisma
model User {
  id               String   @id @default(cuid())
  username         String   @unique
  passwordHash     String   // bcrypt 加密，绝不明文存储
  role             String   @default("user")   // "user" | "admin"
  passwordExpireAt DateTime // 密码过期时间，管理员设置
  isActive         Boolean  @default(true)     // 管理员可一键封禁
  createdAt        DateTime @default(now())
}
```

### 登录校验逻辑

```
用户提交 username + password
  ↓
① 查用户 → 不存在 → 返回"用户名或密码错误"（不透露哪个错）
② bcrypt 比对密码 → 不匹配 → 同上
③ isActive = false → 返回"账号已停用，请联系管理员"
④ passwordExpireAt < now() → 返回"账号已过期，请联系管理员重置"
⑤ 全部通过 → 签发 JWT，有效期7天
```

### 管理员操作（直接操作数据库 or 简单后台页面）

| 操作 | 方式 |
|------|------|
| 创建账号 | 运行脚本 or 后台页面，设置用户名+初始密码+过期时间 |
| 重置密码 | 更新 passwordHash + passwordExpireAt（延期N天）|
| 临时封禁 | isActive = false |
| 查看账号列表 | 后台页面 `/admin/users`（仅 admin 角色可访问）|

### 安全要点

```
✅ 密码用 bcrypt（cost=12），绝不明文/MD5
✅ 登录失败统一提示"用户名或密码错误"，不区分哪个错
✅ 登录接口限流：同IP 5次/15分钟
✅ JWT secret 从环境变量读取，不硬编码
✅ 所有数据接口强制校验 userId = 当前登录用户（防越权）
✅ Admin 路由独立前缀 /admin，中间件验证 role = admin
```

### 3.2 题目入库（默认共享）

```
任何用户录入题目
  ↓
自动按题目内容查重（防重复）
  ↓
不存在 → 入公用库（isPublic=true，默认共享）
         同时绑定到录入者的错题本
  ↓
已存在 → 直接绑定到录入者的错题本，不重复入库

第一个做错该题的用户 → 触发 sharedAiAnalysis 生成（写到 Question 上）
后续所有用户做错同一道题 → 直接读 Question.sharedAiAnalysis，零AI成本

用户可选：关闭某道题的共享（isPublic=false）→ 仅自己可见
```

### 3.3 "笔记→洞察"知识蒸馏机制

> 笔记是个人的，但多个人在同一考点上犯的错误有规律可循。管理员可触发 AI 聚合分析，将共性规律蒸馏为"社区洞察"，匿名公开给所有用户，原始笔记不暴露。

```
用户A的私有笔记："我总把'如果...就'和'只有...才'搞混"
用户B的私有笔记："充分条件经常认反"          ← 普通用户看不到别人的这些
用户C的私有笔记："翻译推理选项A和D老分不清"
         ↓
   [仅 content_manager / admin 可触发]
   POST /api/admin/insights/generate  { skillTag: "翻译推理" }
         ↓
   AI 读取该 skillTag 下所有用户的 UserNote（不含用户标识）
   → 蒸馏出共性规律
         ↓
   生成草稿（isPublished=false）：
   "翻译推理最常见错误：70%用户混淆充分/必要条件方向
    口诀：'如果A则B' → A充分；'只有A才B' → A必要"
   sourceCount=47  （来自47条笔记，不暴露是谁写的）
         ↓
   [管理员审核] → 发布（isPublished=true）
         ↓
   所有用户可见 → GET /api/insights?skillTag=翻译推理
```

**关键安全边界：**
- AI 分析时，笔记内容**不携带 userId**，彻底匿名化后传入
- 发布的洞察只展示"基于 N 条经验"，**原始笔记永远不出现在任何公开接口**
- 用户可设置笔记 `isPrivate=true` → 彻底排除在聚合分析之外

### 3.5 考试目标体系

**两个独立维度，首次登录时必须同时配置：**

| 维度 | 字段 | 说明 |
|------|------|------|
| 考试类型 | `examType` | 决定题目难度、分值标准、出题风格 |
| 分数目标 | `targetScore` | 该考试下的行测目标分（用户自定） |

**三类考试核心差异：**

| 考试 | 枚举值 | 行测题数 | 时间 | 有数量关系 | 参考优秀线 | 难度 |
|------|--------|---------|------|-----------|-----------|------|
| 国家公务员考试 | `guo_kao` | 130题 | 120分钟 | ✅ 15题 | ≥85分 | ★★★★★ |
| 各省公务员考试 | `sheng_kao` | 100-120题（各省不同） | 90-120分钟 | ⚠️ 视省份 | ≥75分 | ★★★☆☆ |
| 多省联考/统考 | `tong_kao` | 100-110题 | 90-100分钟 | ✅ | ≥80分 | ★★★★☆ |

**为什么两个维度必须绑定：**
- 同样"目标75分"，国考极难，省考（简单省份）相对容易 → **不绑定则预测分失真**
- 分值体系不同：各题型每题配分因考试而异 → 影响 §十八 预测分权重
- 题库筛选：练习优先刷与目标考试同类型的真题
- AI出题风格：国考逻辑嵌套复杂，省考相对直接 → Prompt 需感知
- 速度警戒线：国考每题约55秒；省考约50-60秒

## 五、核心练习流程（重构版）

### 5.0 练习模式选择

```
每次进入练习前（或设置中永久选择）：

┌────────────────────┐  ┌────────────────────┐  ┌────────────────────┐
│  ⚡ 快速复习模式    │  │  🎯 深度练习模式   │  │  🔥 同错因聚焦模式  │
│                    │  │                    │  │                    │
│ 不强制写思路        │  │ 必须写思路/涂鸦    │  │ 5-8题同一错因聚焦  │
│ 直接选答案揭晓      │  │ AI验证思维链       │  │ 集中击破一个薄弱点  │
│ 适合：疲惫/快速过   │  │ 适合：专项突破     │  │ 适合：反复栽跟斗时  │
└────────────────────┘  └────────────────────┘  └────────────────────┘

快速模式：修正卡 → 选答案 → 直接揭晓答案+解析
深度模式：修正卡 → 选答案 → 写思路/涂鸦 → 一起揭晓+AI验证
聚焦模式：系统自动按 reasonTag/skillTag 聚合 5-8 道同类错题 → 连续练习同类型

同错因聚焦模式触发逻辑：
  · 用户手动选择：首页"我的薄弱点"→ 选择某个错因分类 → 进入聚焦练习
  · 系统推荐：某 skillTag 下连续答错 ≥ 3 题 → 顶部出现提示"要集中突破吗？"
  · 聚焦练习结束后展示小结：命中率变化 / 思路改善对比 / 建议是否固化新规律

┌────────────────────┐  ┌────────────────────┐
│  🤖 AI变式出题模式  │  │  ✏️ 用户出题模式   │
│                    │  │                    │
│ 同规律/不同领域     │  │ 自己出题让AI批改    │
│ 无限新题不重样      │  │ 能出对题=真理解     │
│ 适合：规律自测通过后 │  │ 适合：深度掌握验证  │
└────────────────────┘  └────────────────────┘

默认使用上次选择的模式，每次可临时切换
```

### 5.1 答题步骤（深度练习模式，修正思维的关键）

```
步骤1：展示修正卡（渐进隐藏）
  根据该题 reviewCount（本题累计复习次数），分3个阶段：

  ── 阶段A（第1-2次复习）：完整展示 ──
  ┌─────────────────────────────────────┐
  │ 上次你的错误                         │
  │ 原因：充分必要条件搞反了              │
  │                                     │
  │ 这次行动规则：                        │
  │ "只有...才..." → 必要条件            │
  │ "如果...则..." → 充分条件            │
  └─────────────────────────────────────┘
  → 用户点击"我记住了"才进入答题

  ── 阶段B（第3-4次复习）：只显示技能点名称 ──
  ┌─────────────────────────────────────┐
  │ 🧠 本题考察：充分必要条件            │
  │ （规则已隐藏，试着回忆再作答）        │
  └─────────────────────────────────────┘
  → 不再给出规则内容，让用户依靠记忆作答

  ── 阶段C（第5次及以后）：进入规律自测模式 ──
  → 修正卡完全隐藏，系统改为主动提问（见 §6.6 规律自测模式）

步骤2：计时
        默认模式：静默记录用时（右上角只显示 ⏱ 秒数，不倒计时不警告）
        严格计时模式（设置中开启）：60秒倒计时 + 边框变红 + 强制提交
        → 默认不开，避免焦虑干扰思考
        → 只有想模拟真实考试时才开严格模式

步骤3：选答案（⚠️ 系统收到答案后不显示对错，静默保存）
  → 用户点击 A/B/C/D → 按钮变为选中态
  → 【不出现"正确/错误"提示】→ 直接进入步骤4

步骤4：按题型写思路（在不知道对错的状态下）
  ├─ 图推/常识/类比 → 无需写思路，直接进步骤5
  │
  ├─ 数量关系/资料分析 → 涂鸦草稿板模式
  │     选完答案后弹出 Excalidraw 草稿板（半屏或全屏）
  │     用户手写/画算式、列式、演算过程
  │     完成后点"提交思路" → 进入步骤5
  │
  └─ 逻辑判断/言语理解/定义判断 → 文字输入模式
        选完答案后弹出文本框
        写逻辑推理链条或解题思路
        完成后点"提交思路" → 进入步骤5

步骤5：【一起揭晓】答案正确/错误 + AI同时评判思路
  ┌───────────────────────────────────────┐
  │ ✅ 答案正确（或 ❌ 答案错误）           │
  │                                       │
  │ 你的思路评分：partial                  │
  │ 💬 方向对，但第2步把充分条件搞反了     │
  └───────────────────────────────────────┘
  → AI验证异步执行，流式逐字出现，不阻塞翻题

步骤6：展示官方解析 + AI首次分析结果 + 可跳转关联笔记
```

**为什么选完答案后不立即揭晓？**
```
选完就揭晓 → 写思路时已知对错 → 会倒推 → 伪思路，暴露不了真实问题
选完后悬念保留 → 写思路时仍不知对错 → 真实推理过程 → AI找到真正断裂点
同时：保留了"先做题"的自然习惯，比"先写才能选"阻力更小
```

### 5.2 严格计时模式下的强制提交（仅设置开启时生效）

```
倒计时从答题开始，60秒归零时（无论用户处于哪个阶段）：

  阶段A：还没选答案时超时
    → isTimeout = true，答案为空，思路为空
    → 直接跳揭晓页，标注"超时未作答"

  阶段B：已选答案，正在涂鸦草稿板时超时
    → canvas.toDataURL('image/png') 自动截图保存
    → isAutoSubmitted = true
    → 立即跳转揭晓页（答案+思路一起揭晓）
    → 后台异步上传图片 + 调AI分析（不阻塞翻题）

  阶段C：已选答案，正在写文字思路时超时
    → 保存当前已输入文字（哪怕不完整）
    → isAutoSubmitted = true
    → 立即跳转揭晓页

  阶段D：已选答案，思路已提交，等待AI结果时超时
    → 不受影响，AI异步返回结果正常展示

结果页统计：
  本轮答题 N 题，超时 N 题，系统强制提交 N 题
```

```
入口：首页 → "快速录错题" 按钮（与"手动录入"并列）

  2. 连续输入（核心交互，每行一题）：
     ┌─────────────────────────────────────────┐
     │ 题号  我的答案  题型            备注（可选）│

     │                                         │
```

### 5.5 AI变式出题模式（解决"换汤不换药"）

```
触发方式：
  ① 规律自测（§6.6）通过后 → 自动提示进入 AI 变式出题
  ② 立即迁移验证（§6.5）→ 题源优先从 AiGeneratedQuestion 取
  ③ 用户主动："给我出5道[充分必要条件]的新题"

生成逻辑（三层输入拼装，见 §十七）：
  层1 静态题型模板 + 层2 用户固化规律 + 层3 随机领域/难度
  → AI 生成 → AI 自检 → 写入 AiGeneratedQuestion

做题体验：
  · 界面与真题完全相同，题目右下角有一个小标记 [AI]
  · 答题后可点 [这道题有问题] → flagCount+1，系统自动接受并重新生成
  · 连续3次被标记 → 该题自动下线（isActive=false）

变式维度（每次随机组合）：
  · 领域：职场 / 家庭 / 医疗 / 法律 / 体育 / 自然科学 / 历史（7选1）
  · 难度：简单（单步）/ 中等（两步）/ 困难（多条件嵌套）
  · 干扰：是否混入1道"形似但实际考察不同规律"的题（辨别训练）
```

## 六、遗忘曲线升降级规则（修订版）

### 6.1 升降级逻辑（2D 结果矩阵）

```
快速复习模式（无思路）：
  答对 → 升级（interval进入下一档）
  答错 → 降级（interval重置为1）

深度练习模式 / 聚焦模式（需写思路，4种状态）：
  ┌───────────────────┬───────────────┬──────────────────────────────────────┐
  │ 状态              │ 思路          │ 升降级结果                            │
  ├───────────────────┼───────────────┼──────────────────────────────────────┤
  │ 状态1：答对        │ correct       │ ✅ 升级（interval进入下一档）真正掌握  │
  │ 状态2：答对        │ partial       │ ⚠️ 维持（答对但思路不稳，继续当前档）  │
  │ 状态3：答对        │ wrong        │ ⚠️ 维持（猜对了，思路根本错，不算掌握）│
  │ 状态4：答错        │ 任意          │ ❌ 降级（interval重置为1）             │
  └───────────────────┴───────────────┴──────────────────────────────────────┘

  ⚠️ 状态2/3 比状态4 更需要警惕：
  能答对但思路错 → 下次遇到变形题会直接暴露，属于"虚假掌握"

⏱ 速度维度（叠加到4种状态上）：
  状态1 + 用时 ≤ 速度警戒线 → 正常升级（interval进入下一档）
  状态1 + 用时 > 速度警戒线 → "慢正确"：isSlowCorrect=true，interval 不升档
                               （答对但实考会超时，不算真掌握）
  速度警戒线默认值（用户可调整）：
    资料分析 180秒/题 · 逻辑判断 90秒/题 · 言语理解 45秒/题
    数量关系 210秒/题 · 常识判断 30秒/题

⚠️ "升半档"的具体实现（[1,3,7,15,30]序列无法取半档，统一约定为"不升档"）：
  isSlowCorrect=true 时：
    · reviewInterval 保持原值（不推进到下一档）
    · masteryPercent 惩罚 -10%（在 §6.3 updateMastery 中处理）
    · 下次复习仍按原 interval 触发，给用户一次在限时内做对的机会
    · 实现代码：if (record.isSlowCorrect) { /* interval 不变，mastery -10 已在updateMastery处理 */ }
    · ≠ 降级（interval不重置为1）；≠ 正常升级（interval不推进）

🔥 易错标记（isHot）：
  · 连续2次 状态4（答错）→ isHot = true
  · 连续2次 状态2或3（思路有问题）→ isHot = true（"答对但思路不稳"同样危险）
  · 连续3次 isSlowCorrect → isHot = true（反复慢正确，需要专项提速训练）

ReviewRecord 2D字段：
  isCorrect       Boolean   // 答题对/错（唯一字段名，旧称 answerCorrect 已废弃）
  thinkingVerdict String?   // correct/partial/wrong/null（null=快速模式）
  resultMatrix    String?   // "1"=升级/"2"=维持(答对思路partial)/"3"=维持(答对思路错)/"4"=降级
  isSlowCorrect   Boolean   // 答对但超速度警戒线（影响升半档规则）
```

### 6.2a masteryPercent 初始值与首次更新（⚠️ 开发必读）

> **问题**：Schema 定义 `masteryPercent Int @default(0)`，但 errorROI 公式中 0-30% 区间的 stockifyProb=0.1，意味着新录入的题会被长期压在推题队列底部，与"先打高ROI题"的设计目标冲突。

**规则（全系统统一）：**

```typescript
// 创建 UserError 时（第一次录入/第一次答错）：
//   masteryPercent = 0  ← 初始值，还没复习过，不具备mastery评估
//   reviewInterval  = 1 ← interval序列起点
//   nextReviewAt    = tomorrow

// 第一次复习后（updateMastery 首次执行）：
//   答对 + 思路correct → mastery = intervalToMastery[1] + 5 = 25
//   答对 + 思路partial → mastery = intervalToMastery[1]     = 20
//   答对 + isSlowCorrect → mastery = 20 - 10               = 10
//   答错               → mastery = max(0 - 30, 10)         = 10

// ⚠️ errorROI 排序时对初始态（masteryPercent=0）的处理：
//   新录入题 masteryPercent=0，按公式属于 0-30% 区间（stockifyProb=0.1）
//   → 合理：真的刚录入，还没建立任何mastery，低优先级符合预期
//   → 第一次复习后 mastery=20-25，仍在0-30%区间，属"建设中"状态
//   → 约第3-4次复习后进入30-60%区间（stockifyProb=0.4），开始进入主力队列
//   → 这个自然提升路径不需要人工干预，初始值0无需调整
```

> **结论**：`masteryPercent=0` 的初始值是正确的，不要在创建 UserError 时预设一个非零值。新题低优先级是设计预期，让用户先把"快到手的"题存量化，再回来攻新题。

### 6.3 masteryPercent 更新公式（每次答题后执行，全系统唯一定义）

> ⚠️ **本节是整个系统最关键的计算规则**：masteryPercent 驱动 errorROI、四状态机、增量候选识别。必须在每次 ReviewRecord 写入后立即异步执行。

```typescript
// 每次答题结束后，异步更新 UserError.masteryPercent
function updateMastery(userError: UserError, record: ReviewRecord): number {
  // Step 1：根据当前 interval 档位确定基础 mastery
  const intervalToMastery: Record<number, number> = { 1: 20, 3: 40, 7: 60, 15: 75, 30: 95 }
  const base = intervalToMastery[userError.reviewInterval] ?? 20

  // Step 2：质量修正
  let mastery = base
  if (record.thinkingVerdict === 'correct') mastery += 5   // 思路完全正确 +5%
  if (record.isSlowCorrect === true)        mastery -= 10  // 慢正确：实考超时，-10%
  mastery = Math.min(100, Math.max(0, mastery))            // 夹到 [0, 100]

  // Step 3：答错时重置（不归零，保留底部记忆）
  if (!record.isCorrect) {   // ⚠️ 用 isCorrect，旧称 answerCorrect 已废弃
    mastery = Math.max(userError.masteryPercent - 30, 10)
    // interval 已在升降级逻辑中重置为 1，此处不重复
  }

  return mastery
  // → 调用方MUST按以下顺序执行（顺序严格，不可颠倒）：
  //   1. 写入 UserError.masteryPercent = newMastery
  //   2. ⚠️ 先 push 到 masteryHistory（JSON.stringify([...history, newMastery])）
  //   3. 再调用 checkStockified(userError)  ← 依赖已更新的 history（见 §6.3c）
  //   4. 调用 updateDecayRate / updateStabilityScore（见 §6.3a/b）
  //   5. 写入 userError.lastReviewedAt = new Date()（见 §6.3a 末尾注释）
}
```

**关键原则：** 答错不清零（防一次手误抹掉几周积累）；慢正确有惩罚（实考限时场景下慢正确 ≠ 真掌握）；interval 档位决定 base，mastery 是可观测置信度映射。

### 6.3a decayRatePerDay 更新公式（每次答题后执行）

```typescript
function updateDecayRate(userError: UserError, newMastery: number): number {
  const history: number[] = JSON.parse(userError.masteryHistory ?? '[]')
  if (history.length < 2) return userError.decayRatePerDay  // 历史不足，暂不更新

  const prevMastery = history[history.length - 1]   // 上次复习后的 mastery
  const masteryDrop = prevMastery - newMastery       // 遗忘导致的下降值

  // ⚠️ 用实际经过天数，而非 reviewInterval（计划天数）
  // lastReviewedAt 由 Schema 新增字段提供（见 §0.3 Schema），每次答题后同步写入
  const daysBetween = userError.lastReviewedAt
    ? differenceInDays(new Date(), userError.lastReviewedAt)
    : userError.reviewInterval  // 首次退化：lastReviewedAt 为空，回退到计划值

  if (masteryDrop <= 0 || daysBetween <= 0) return 0.0

  return Math.max(0, masteryDrop / daysBetween)
  // §0.5 筛选条件：decayRatePerDay < 0.05
  // 含义：每天遗忘不超过0.05百分点 → 曲线平缓 → 考前激活有效
}
// ⚠️ 调用本函数后，MUST 同步更新：userError.lastReviewedAt = new Date()
```

### 6.3b stabilityScore 更新公式（每次答题后执行）

> **使用场景**：§16.3 守队列排序——`stabilityScore 升序`（最低稳定性的已存量化题优先复习）。
> 数值含义：越小 = 近3次mastery波动越大 = 越不稳定 = 越需要优先加固。
> 不用于 isStockified 判断（那里用 masteryHistory 直接比较相邻值）。

```typescript
function updateStabilityScore(masteryHistory: number[]): number {
  const recent = masteryHistory.slice(-3)

  // ⚠️ 边界条件处理（开发时容易漏）：
  //   历史记录 < 2 条（reviewCount=0或1）→ 无法计算方差，返回默认中等稳定值
  //   历史记录 = 2 条 → 可计算，但样本太少；结果仅供参考，不影响 isStockified 判断
  //   历史记录 ≥ 3 条 → 正常计算
  if (recent.length < 2) return 50.0   // 历史不足，默认中等稳定（不影响存量化判断）

  const mean     = recent.reduce((a, b) => a + b, 0) / recent.length
  const variance = recent.reduce((sum, x) => sum + Math.pow(x - mean, 2), 0) / recent.length

  // ⚠️ 防除零：variance=0 说明近3次完全一致 → 视为最高稳定性
  return variance === 0 ? 100.0 : 1 / variance
  // 注意：返回值是相对量（1/variance），仅用于相互排序，不代表绝对百分比
}

// stabilityScore 与 isStockified 的独立性说明：
//   stabilityScore 低 ≠ 不能存量化（checkStockified 有自己的判断逻辑，见§6.3c）
//   stabilityScore 只影响"已存量化题的守卫队列复习顺序"
//   两者互不干扰，不要混用
```

### 6.3c checkStockified 判定（每次 masteryHistory 更新后执行）

> **执行时机**：MUST 在 masteryHistory 已 push 新值后调用（见 §6.3 末尾注释）。
> 若在 push 之前调用，`history[-1]` 仍是旧值，"近2次未降>10%"判断将基于过期数据。

```typescript
function checkStockified(userError: UserError): boolean {
  // 条件1：mastery 达到存量化门槛
  if (userError.masteryPercent < 80) return false

  // 条件2：最近一次非慢正确（慢正确不算真掌握）
  // isLastSlowCorrect 由 ReviewRecord.isSlowCorrect 写入时同步维护
  if (userError.isLastSlowCorrect) return false

  // 条件3：近2次 mastery 均未下降超过10%（需至少3条历史记录，才能检验2组相邻变化）
  const history: number[] = JSON.parse(userError.masteryHistory ?? '[]')
  if (history.length < 3) return false  // 数据不足（需3条才能验证2次变化），视为未稳定

  const [p2, p1, curr] = history.slice(-3)  // ⚠️ 依赖 push 后的 history
  const drop1 = p2 - p1    // 倒数第2次的 mastery 变化量
  const drop2 = p1 - curr  // 倒数第1次的 mastery 变化量
  if (drop1 > 10 || drop2 > 10) return false  // 任意一次下降>10%，视为不稳定

  return true
}
// 返回 true → 更新 UserError.isStockified = true，触发记忆锚点生成（§0.7⑤）
// 反弹判断：若 isStockified=true 但 masteryPercent < 60% → 重置为 false，降回增量候选
```

### 6.8 复习队列溢出管理（防第 2 周放弃）

```
问题场景：
  日常录题 10 题/天，每题到期复习 → 第2周队列可能积压到 40-60 题
  用户看到"待复习 63 题" → 直接关 APP → 永久放弃

三条管理规则：

策略1：每日软上限（防止单日积压感）
  · 每日复习上限：min(dailyGoal × 1.5, dailyGoal + 30)
    // dailyGoal=70时上限约100题；×1.5取整保留弹性但不超量
    // ⚠️ 不要写"默认20题"——那是 dailyGoal=10 时的值，与系统设计基准 dailyGoal=70 矛盾
  · 超出部分不显示为"今天待做"，而是"明日优先"
  · 每日完成上限后，状态变为"今日收工 ✅"（不再显示更多）
  · 目的：用户每天看到的是"有限的、可完成的"队列，而不是无底洞

策略2：积压预警与清欠模式
  积压题数（超期未做 ≥ 2 天）触发：
    · 5-10题积压  → 首页轻提示（不突出）
    · 11-20题积压 → 橙色警告："有 N 题等了 2 天以上，今天多做 5 题？"
    · 21-30题积压 → 红色提示 + 推荐"30分钟清欠模式"
    · > 30题积压  → 暂停今日精选推送，只消化存量；用户确认后才恢复新题推送

  清欠模式：
    · 一次性呈现 10 道最老的积压题（interval 最小的优先）
    · 全部使用"快速复习模式"（不强制写思路）
    · 目标：30 分钟内清掉一批，让队列回到可控范围

策略3：智能分流（考试日期临近时）
  距考试 ≤ 30 天 → 新题录入提示："时间有限，建议优先消化存量错题"
  距考试 ≤ 14 天 → 系统自动冻结新题录入（仍可手动解锁）
  距考试 ≤ 7 天  → 全部切换快速复习模式，不再推深度练习
```

```
  · 每次导入最多调用AI修复3题（cost保护）
```

### 7.4 固化思维（两种路径）

**路径A：快速固化（主路径，AI首次分析完成后自动出现）**

```
AI分析结果展示区底部，直接出现：

  ┌─────────────────────────────────────────┐
  │ 📌 固化这条规律                          │
  │ AI建议：看到"只有...才..."→ 判断必要条件  │
  │                                         │
  │ 你的版本（修改后才能保存）：             │
  │ [________________________]              │
  │                    [保存到我的规律库]    │
  └─────────────────────────────────────────┘

  · 输入框不预填内容（防止用户无脑复制AI错误结论）
  · AI建议显示在上方仅供参考
  · 用户输入后才激活"保存"按钮
  · 操作路径：看分析 → 10秒内完成固化，零打断
```

**路径B：深度讨论（可选，用户主动触发）**

```
System Prompt：
  你是公务员考试解题思维教练，目标是帮用户发现可复用的规律。
  1. 先倾听疑问，不急着给答案
  2. 用追问引导用户自己发现（苏格拉底式）
  3. 收敛时说"我们固化一下：看到XX→做YY，这样准确吗？"
  4. 用户修正你的草稿时，以用户版本为准

触发入口：
  · 错题详情页 → "💬 想深入讨论这道题"（次要按钮，不突出）
```

### 7.5 费用控制

```
AI调用类型（仅3种，其余全部脚本/DB替代）：
  ① 首次录入分析：1次/题，永久缓存
  ② 思路验证：每次复习需写思路时1次
  ③ 讨论对话：用户主动触发，平均每次3-5轮
  （PDF解析 → Python脚本，搜索 → PostgreSQL，重复检测 → Python）

估算（每天完成70道，其中新录入约5题做首次AI分析，错题复习约10题需写思路走AI验证）：
  // 系统默认总量70道/天（错题优先30道 + 真题补位40道）；AI仅对需写思路的深度复习触发
  首次分析 5题（新录入）：5 × ¥0.003 = ¥0.015/天
  思路验证 10次（深度复习中约1/3题写思路）：10 × ¥0.002 = ¥0.02/天
  讨论对话（偶发）：~¥0.01/天
  合计：约 ¥1.5-3/月（低于原预算）
```

## 十三、PDF试卷导入

### 13.1 完整流程

```
1. 用户点击「导入PDF试卷」→ 选择文件
   安全校验（前后端双重）：
   ├─ 文件大小 ≤ 10MB
   ├─ 扩展名必须是 .pdf
   └─ 服务端验证 magic bytes：前4字节必须是 %PDF（25 50 44 46）

2. 服务端调用 Python 脚本 parse_pdf.py
   ├─ pdfplumber 提取纯文本
   ├─ 正则引擎识别题号/选项/答案/解析
   ├─ 每题打 confidence 分（0-1）
   └─ 失败（图片型PDF）→ 提示"此PDF为扫描图片，请使用可复制文字的PDF"

3. AI兜底（仅对 confidence < 0.6 的单题）：
   → 取该题原始文本片段 → MiniMax修复单题结构
   → 每次导入 AI 兜底上限：3题（防止cost失控）

4. 确认页（只展示异常题，正常题静默自动入库）：

   · confidence ≥ 0.75 的题目 → 直接入库，不打扰用户
   · confidence < 0.75 的题目 → 弹出确认：

   ┌──────────────────────────────────────────┐
   │ ✅ 23题已自动导入                         │
   │                                          │
   │ ⚠️ 2题需要你确认（置信度低）：            │
   │ [题目2] 数量关系·未识别    [选分类▾] [删除]│
   │ [题目7] 选项残缺，建议手动录入 [编辑] [删除]│
   │                                          │
   │             [完成导入]                   │
   └──────────────────────────────────────────┘

   → 大多数情况用户只需点"完成导入"，无需逐题审查

5. 用户确认 → 批量写入 Question 表
   ├─ 每题先用 Python difflib 查重（文本相似度>85%即视为重复）
   ├─ 不重复 → 入公用库 + 绑定到导入者
   └─ 重复 → 跳过，提示"N题已存在，跳过"
```

### 13.2 Python 解析脚本（scripts/parse_pdf.py）

```python
# scripts/parse_pdf.py
# 无AI，纯正则，覆盖公考真题标准格式
import pdfplumber, re, json, sys

QUESTION_RE = re.compile(r'(?:^|\n)(\d{1,3})[.、．]\s*(.+?)(?=\n\d{1,3}[.、．]|\Z)', re.S)
OPTION_RE   = re.compile(r'([ABCD])[.、．]\s*(.+?)(?=[ABCD][.、．]|\n|\Z)')
ANSWER_RE   = re.compile(r'(\d+)[.、．]\s*([ABCD])')  # 答案页格式

def parse(pdf_path: str) -> list[dict]:
    with pdfplumber.open(pdf_path) as pdf:
        text = "\n".join(p.extract_text() or "" for p in pdf.pages)

    results = []
    for m in QUESTION_RE.finditer(text):
        q_text   = m.group(2).strip()
        options  = [f"{k}.{v.strip()}" for k, v in OPTION_RE.findall(q_text)]
        clean_q  = OPTION_RE.sub("", q_text).strip()
        confidence = 1.0 if len(options) == 4 else (0.5 if options else 0.2)
        results.append({
            "question": clean_q,
            "options":  options or None,
            "answer":   None,       # 答案单独从答案页提取
            "type":     None,       # 由用户在预览页确认
            "confidence": confidence,
        })
    return results

if __name__ == "__main__":
    print(json.dumps(parse(sys.argv[1]), ensure_ascii=False))
```

### 13.3 服务端接口安全要点

```typescript
// - 存 /tmp 临时目录，Python 处理完立即 fs.unlinkSync 删除，不长期保留
// - 子进程调用 Python，stdout 解析 JSON，stderr 写日志不暴露给前端

```

```sql
-- SQL migration（一次性，在 Supabase SQL Editor 执行）
-- 添加全文检索列（中文用 simple 分词器，支持字符级匹配）
ALTER TABLE questions
  ADD COLUMN search_vector tsvector
    GENERATED ALWAYS AS (
      to_tsvector('simple',
        coalesce(question,'') || ' ' ||
        coalesce(analysis,'') || ' ' ||
        coalesce(subtype,'') || ' ' ||
        coalesce(sub2,''))
    ) STORED;

CREATE INDEX idx_questions_fts ON questions USING gin(search_vector);
```

```sql
-- 搜索接口（Next.js API Route 中执行，Prisma $queryRaw）
SELECT id, question, type, subtype, sub2,
       ts_rank(search_vector, query) AS rank
FROM questions,
     plainto_tsquery('simple', $1) AS query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 20;

-- 无需 AI，无需 embedding，响应 < 50ms
```

### 14.2 重复检测（Python脚本，无需AI）

```python
# scripts/dedup_check.py
from difflib import SequenceMatcher

def is_duplicate(new_q: str, existing_questions: list[str], threshold=0.85) -> bool:
    """文本相似度 > 85% 视为重复，防止同一题多次入库"""
    for eq in existing_questions:
        ratio = SequenceMatcher(None, new_q.strip(), eq.strip()).ratio()
        if ratio > threshold:
            return True
    return False
```

## 十五、存储层分离（Repository Pattern）

### 设计原则
> 业务逻辑不直接依赖 Prisma，只依赖 Repository 接口。
> 未来换数据库（如 MongoDB/MySQL）只需换 Repository 实现，Service 层零改动。

```typescript
// lib/repositories/prisma/QuestionRepositoryPrisma.ts
import { prisma } from '@/lib/db'
import type { QuestionRepository } from '../interfaces'

export class QuestionRepositoryPrisma implements QuestionRepository {
  async search(keyword: string, limit = 20) {
    // PostgreSQL 全文检索
    return prisma.$queryRaw`
      SELECT * FROM questions
      WHERE search_vector @@ plainto_tsquery('simple', ${keyword})
      ORDER BY ts_rank(search_vector, plainto_tsquery('simple', ${keyword})) DESC
      LIMIT ${limit}
    `
  }

  async checkDuplicate(questionText: string) {
    // 精确查重（全文检索初步筛选，Python difflib 二次确认）
    return prisma.question.findFirst({
      where: { question: { contains: questionText.slice(0, 20) } }
    })
  }
  // ... 其他方法
}
```

### 16.7 targetRate 动态计算（从 targetScore 反算各题型目标）

```

  │ 资料分析   │ 1.08     │ baseRate × 1.08（略高要求）│
  │ 数量关系   │ 0.82     │ baseRate × 0.82（容许更低）│
  │ 常识判断   │ 0.88     │ baseRate × 0.88          │
  └──────────────────────────────────────────────────┘
  // 难度系数反映：资料分析更可通过练习提升，常识靠积累难以快速突破

  示例（国考，targetScore=85）：
    言语理解   目标：85%
    判断推理   目标：78%
```

### 16.8 战略建议：何时应该战略跳过某考点

```
触发条件（存量化框架下的战略跳过逻辑）：

  情况A：某考点 masteryPercent < 30% 且 daysToExam ≤ 30天
    → timeViability 接近 0，存量化来不及
    → 提示："⚠️ 距考试不足30天，该考点从0建立存量化需要5-8轮，
             建议考前不再主攻，靠考前临时复习抢一点即可"

  情况B：某题型所有考点已全部 isStockified=true
    → 进入"维持模式"：只进守队列，不进攻队列
    → 提示："✅ 资料分析所有考点已存量化！已切换为维持模式"

  情况C：daysToExam ≤ 14天（临考期）
    → timeViability=0，所有 masteryPercent < 70% 的考点自动移出攻队列
    → 全力巩固已接近存量化的考点，不开拓新考点

用户可选择：
  [接受建议] → 该考点 errorROI 归零，移出今日任务攻队列
  [坚持练]   → 维持原有推荐，系统不再提醒

注：战略跳过不删除错题，不影响间隔复习；只是攻队列不主动推
```

### 17.3 pgvector RAG检索流程

```sql
-- Step 1：将用户录入规律时的 finalContent 向量化并存储
-- 触发时机：UserInsight.finalContent 保存/更新时
-- 调用：MiniMax embedding API → 生成 1536维向量 → 存入 embedding 列

UPDATE user_insights
SET embedding = '[0.023, -0.415, ...]'::vector
WHERE id = $insightId;

-- Step 2：AI出题时，检索当前用户最相关的规律（余弦相似度）
SELECT id, skill_tag, final_content, ai_gen_count
FROM user_insights
WHERE user_id = $userId
  AND is_active = true
  AND skill_tag = $targetSkillTag
ORDER BY embedding <=> $queryEmbedding::vector
LIMIT 3;

-- Step 3：将检索到的 finalContent 拼入 Prompt 第二层
-- （见 §17.4 Prompt模板）
```

> **零运维**：用户每次保存规律时自动触发向量化（异步任务队列）。Supabase 免费套餐内置 pgvector 扩展，无需额外部署。

### 18.2 分值权重差异（影响预测分计算）

预测分公式：
```
predictedScore = Σ(题型正确率 × 题型满分分值)
```

各题型满分分值（`scoreWeight`，按考试类型注入 statsJson）：

| 题型 | 国考（参考） | 省考通用（参考） | 统考（参考） |
|------|------------|--------------|-----------|
| 言语理解 | 32分 | 35分 | 32分 |
| 判断推理 | 36分 | 30分 | 32分 |
| 资料分析 | 20分 | 20分 | 20分 |
| 数量关系 | 15分 | 0-10分（视省份） | 10分 |
| 常识判断 | 12分 | 15分 | 12分 |
| **行测满分** | **~100分** | **~100分** | **~100分** |

**实现方式**：
```typescript
// lib/scoring/weights.ts
export const SCORE_WEIGHTS: Record<string, Record<string, number>> = {
  guo_kao:   { 言语理解: 32, 判断推理: 36, 资料分析: 20, 数量关系: 15, 常识判断: 12 },
  sheng_kao: { 言语理解: 35, 判断推理: 30, 资料分析: 20, 数量关系: 0,  常识判断: 15 }, // 默认无数量关系
  tong_kao:  { 言语理解: 32, 判断推理: 32, 资料分析: 20, 数量关系: 10, 常识判断: 12 },
}

export function getWeights(examType: string, targetProvince?: string) {
  // 省考：广东/北京等有数量关系的省份特殊处理
  const HAS_MATH_PROVINCES = ['广东', '北京', '上海', '浙江']
  if (examType === 'sheng_kao' && targetProvince && HAS_MATH_PROVINCES.includes(targetProvince)) {
    return { ...SCORE_WEIGHTS.sheng_kao, 数量关系: 10 }
  }
  return SCORE_WEIGHTS[examType] ?? SCORE_WEIGHTS.guo_kao
}
```

`< 60分` → 顶部橙色提示条：「完善设置，让预测分更准确」

### 19.2 错题数据质量评估

各题型有效预测所需最低样本量：

| 题型 | 最低样本 | 不足时的处理 |
|------|---------|-----------|
| 言语理解 | ≥20题 | 预测分标注"⚠️ 样本不足，仅供参考" |
| 判断推理 | ≥15题 | 同上 |
| 资料分析 | ≥10题 | 同上 |
| 数量关系 | ≥8题 | 同上（国考/统考用户） |
| 常识判断 | ≥10题 | 同上 |

**速度健康度警告**：
- `isSlowCorrect` 比例 > 30%（某题型）→ 仪表盘提示"答对但偏慢，实考相当于丢分"
- 今日精选自动增加该题型严格计时题比重（触发限速训练）

### 19.3 存量化进度可信度分级

| 等级 | 条件 | 进度误差 | 含义 |
|------|------|---------|------|
| 🟢 高可信 | 所有题型样本充足 + ≥2次真实模拟成绩 | ±3% | 进度数字基本可信 |
| 🟡 中可信 | 部分题型不足 或 仅1次模拟成绩 | ±8% | 仅供参考，继续练习会更准 |
| 🔴 低可信 | 多题型样本不足 + 无模拟成绩 | ±17% | 优先补充数据，进度数字为估算 |

首页展示格式：
```
存量化进度：72%（🟡 中可信，±8%）
  点击"?"展开：
  · 资料分析：已稳固 4/6 个考点 ✅
  · 判断推理：已稳固 2/8 个考点 — 主要努力方向
  · 言语理解：⚠️ 样本不足（仅8题），进度估算偏低
  · 数量关系：已稳固 1/5 个考点（建议考虑战略跳过）
  · 常识判断：已稳固 2/4 个考点 ✅
```

> 注：±X% 的直觉感受：±8% 意味着"显示72%时，实际在64-80%之间"——相比旧版±7分，
> 这种相对误差给用户的方向感依然正确，焦虑感更低。

## 二十一、真题语料库与考点图谱

> 历年真题是最权威的"出题频率地图"——系统从中提取 ExamTopicStats，让推荐从"猜"变"算"。

### 21.1 历年真题导入管道

```
真题来源（支持三种格式）：
  · PDF 试卷 → OCR 解析 → 结构化 JSON
  · Excel/Word 整理版 → 直接 import
  · 第三方题库 API（Phase 2）

导入字段映射：
  Question.examType           ← 来源考试（guo_kao/sheng_kao/tong_kao）
  Question.srcProvince        ← 省考填省份（如"广东"），国考/统考填 null
  Question.srcExamSession     ← "2024年国考第一批"
  Question.isFromOfficialBank ← true
  Question.skillTag           ← AI 自动标注 + 人工校对
  Question.difficulty         ← AI 基于正确率估算（easy/medium/hard）
  Question.estimatedCorrectRate ← 历史考生平均正确率（若有来源数据）

去重策略（防止同一道题重复入库）：
  1. 题面文本 MD5 相同 → 跳过，仅更新 srcExamSession
  2. 语义相似度 ≥ 0.95（bge-m3 向量）→ 标记为"相似题"，人工确认后合并
  3. 同 srcExamSession + 题号相同 → 强制覆盖（补录更正）
```

### 21.2 ExamTopicStats 设计与预留口子

ExamTopicStats 表的核心设计思路：**每个 (考试类型, 省份, 考点)** 是一个独立统计单元。

```
预留口子说明：

  国考：srcProvince = null
    → 无省份区分，全国统一一套题库

  统考：srcProvince = null（或填具体省份，因部分统考各省卷不同）
    → Phase 2 再拆分

  省考：srcProvince = "广东" / "浙江" / "江苏" ... （34个省市）
    → 每省独立统计，互不干扰
    → 某省无数据时降级：省级 → sheng_kao通用 → guo_kao
    → 新增一个省份：直接插入新行，无需改表结构
    → 扩展到地市联考：srcProvince 填 "广东-广州" 精确到市即可

  考点标签(skillTag)对齐规则：
    ExamTopicStats.skillTag = Question.skillTag（枚举对齐，不随意新增）
    导入遇到未知考点 → 写入 pendingSkillTag 字段 → 人工归并后批量更新
```

**频率计算（异步 Job，每次真题导入后触发）：**

```typescript
async function rebuildFrequency(examType: string, srcProvince: string | null) {
  const total = await prisma.question.count({
    where: { examType, srcProvince, isFromOfficialBank: true }
  });
  const byTag = await prisma.question.groupBy({
    by: ['skillTag'],
    where: { examType, srcProvince, isFromOfficialBank: true },
    _count: { id: true }
  });
  for (const row of byTag) {
    await prisma.examTopicStats.upsert({
      where: { examType_srcProvince_skillTag: { examType, srcProvince, skillTag: row.skillTag } },
      update: { totalCount: row._count.id, frequency: row._count.id / total },
      create: {
        examType, srcProvince, skillTag: row.skillTag,
        sectionType: lookupSectionType(row.skillTag),
        totalCount: row._count.id, frequency: row._count.id / total
      }
    });
  }
}
```

### 21.3 考点盲区检测


### 21.4 冷启动推荐队列

> 新用户无错题时，errorROI 公式无法运算，用真题频率图谱替代

```
冷启动逻辑：

  coldStartScore = ExamTopicStats.frequency
                 × (1 - ExamTopicStats.avgDifficulty + 0.3)  // 偏中等难度
                 × sectionScoreWeight                         // 高分值题型优先

ELSE IF userErrorCount < 10:

    有错题的题型 → errorROI 排序
    无错题的题型 → coldStartScore 补充
  合并后取 TOP 5

  练完后切换：  "已有 10 条错题，开始个性化推荐 🎯"
```

### 21.5 errorROI 精修到 skillTag 粒度

v7.7 将 errorROI 完整升级为"存量化价值"框架，加入skillFrequencyWeight：

```
errorROI 完整公式（v7.7·存量化框架）：
  errorROI = stockifyProb             // 存量化维度：今天练能推过门槛的概率
           × stockifyValue            // 价值维度：一旦存量化能贡献多少稳固分
           × timeViability            // 时间维度：距考试越近低掌握度越不值得
                                      //   其中 stockifyValue = scoreWeight × skillFrequencyWeight

各因子范围：
  stockifyProb    : 0.1 / 0.4 / 0.8（按masteryPercent分段）
  stockifyValue   : [0, scoreWeight × 3.0]（skillFrequencyWeight上限3.0）
  timeViability   : 0.0 / 0.1 / 0.2 / 1.0（按daysToExam+masteryPercent分段）

skillFrequencyWeight 对比示例（判断推理·国考）：
  翻译推理：frequency=19%，平均=10% → weight=1.9
  朴素逻辑：frequency= 5%，平均=10% → weight=0.5
  同等 stockifyProb 下，翻译推理 errorROI 是朴素逻辑的 3.8 倍
  → 今日任务优先推翻译推理，与真实考试重要性一致 ✅
```
