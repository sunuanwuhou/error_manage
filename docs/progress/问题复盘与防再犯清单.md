# 问题复盘与防再犯清单

> 作用：把真实踩过的坑沉淀成团队共享经验，避免同类错误反复出现  
> 更新规则：只记录“真实发生过、影响过交付、值得以后避免”的问题

---

## 使用规则

以下情况必须新增或更新一条记录：

- 某个问题已经真实影响了开发、测试、交付或用户体验
- 某个问题属于“第二次出现”或“容易重复出现”
- 某个问题的修复动作本身值得变成团队规则

不需要记录的情况：

- 纯临时 typo
- 没有真实影响的探索性失败
- 已知且显然的一次性本地操作失误

每条记录尽量回答 5 件事：

1. 现象是什么
2. 真正根因是什么
3. 当时是怎么修的
4. 以后怎么更早发现
5. 以后怎么避免再犯

---

## 记录模板

复制下面模板追加：

```md
## [编号] 标题

- 时间：
- 场景：
- 现象：
- 根因：
- 修复：
- 预警信号：
- 防再犯规则：
- 关联文件：
```

## [001] 不能拿 build 通过代替页面可用

- 时间：2026-03-19
- 场景：开发服务与页面可用性验证
- 现象：
  - `npm run build` 通过
  - 终端出现 `Ready`
  - 但浏览器里仍可能看到裸 HTML、样式缺失或旧页面残留
- 根因：
  - 构建成功只能说明产物可生成，不能说明浏览器实际命中的是新服务
  - `dev:restart` / 本地残留进程 / 端口漂移会让浏览器命中旧服务
- 修复：
  - 在 `AGENTS.md` 中提升“真实页面可用”的优先级
  - 补充端口监听、真实命中进程、页面渲染状态三重验证
- 预警信号：
  - 日志显示 `Ready`
  - 但页面视觉明显不符合当前代码
  - PID、日志、端口三者不一致
- 防再犯规则：
  - 不允许再用“build 通过”直接宣布可运行
  - 至少要额外验证一次真实页面渲染结果
- 关联文件：
  - `AGENTS.md`
  - `scripts/restart-dev.sh`
  - `docs/progress/current_snapshot.md`

## [002] dev:restart 的 PID / 日志 / 真实服务可能不一致

- 时间：2026-03-19
- 场景：本地开发服务重启
- 现象：
  - `.dev-server.log` 里显示 `Local: http://localhost:3000`
  - `.dev-server.pid` 有值
  - 但真实进程可能已经退出，或 3000 上挂的是旧残留服务
- 根因：
  - 旧版本重启脚本通过 `npm run dev` 后台启动，记录到的可能是包装进程，不是长期存活的 Next 进程
  - 本地环境里旧 node 监听进程未清干净时，浏览器会命中错误服务
- 修复：
  - 重启脚本改为直接启动 `next dev`
  - 验证时同时检查端口监听与实际服务来源
- 预警信号：
  - `Ready` 后端口仍不可达
  - PID 文件中的进程不存在
  - 用户页面与当前代码完全不符
- 防再犯规则：
  - `dev:restart` 只能作为“尝试启动”，不能作为“验证通过”
  - 需要把“端口监听 + 页面实际命中”作为独立检查项
- 关联文件：
  - `scripts/restart-dev.sh`
  - `AGENTS.md`

## [003] 浏览器旧 session 里的 userId 可能和本地数据库脱节

- 时间：2026-03-20
- 场景：onboarding / settings / 任何依赖 `session.user.id` 的用户态接口
- 现象：
  - 页面已登录
  - 提交时出现“保存失败”或 `No User found` / `Record to update not found`
- 根因：
  - 本地数据库重建、数据恢复或账号重建后，浏览器 JWT 里仍保留旧 userId
  - 前端看起来已登录，但后端写入目标用户已经不存在
- 修复：
  - 在 `auth.ts` 里基于 `token.name` 做 userId 自愈回填
  - 在关键接口里把 `P2025` 这类错误转成明确提示，而不是黑盒失败
- 预警信号：
  - 登录正常，但所有用户态写接口都失败
  - 错误集中出现在 onboarding / settings / review submit 等更新接口
- 防再犯规则：
  - 本地数据库有重建或恢复动作后，优先怀疑 session 漂移
  - 涉及 `session.user.id` 的关键写接口，尽量提供明确错误文案
- 关联文件：
  - `src/lib/auth.ts`
  - `src/app/api/onboarding/route.ts`
  - 其他依赖 `session.user.id` 的用户态接口

## [006] Quick Tunnel 地址不能只靠人工翻日志

- 时间：2026-03-20
- 场景：Docker 开发挂载 + 外网分享调试
- 现象：
  - `cloudflared` 每次启动都会生成新的 `trycloudflare.com` 地址
  - 如果地址只存在容器日志里，下一次会话很容易忘记去哪里看
  - 用户明明已经启动成功，却还要再次询问“当前域名是多少”
- 根因：
  - Quick Tunnel 天生是临时域名
  - 之前缺少“启动后自动落盘 + 页面可见”的固定出口
- 修复：
  - `tunnel` sidecar 启动后把最新地址写入 `.runtime/tunnel-url.txt`
  - 登录页直接读取并展示当前外网地址
  - README 和快照补充统一启动命令与查看路径
- 预警信号：
  - 启动成功但团队成员仍在反复问 tunnel 域名
  - 需要人工翻 `docker compose logs tunnel` 才知道当前地址
- 防再犯规则：
  - 临时域名类运行时信息，必须有固定落点，不允许只存在日志里
  - 能在主流程页面展示的运行时地址，不要要求用户自己去查容器日志
- 关联文件：
  - `docker-compose.dev.yml`
  - `Dockerfile.cloudflared`
  - `docker/cloudflared-entrypoint.sh`
  - `src/app/login/page.tsx`
  - `tsc` 又把 `.next/types` 当成 include 输入，导致假失败级联
- 修复：
  - 清理 `.next`
  - 从干净状态重新执行 `build`
  - 再复跑 `tsc`
- 预警信号：
  - 错误同时涉及大量 `.next/types/...`
  - 缺失的是生成产物，不是源码文件
- 防再犯规则：
  - 如果错误集中在 `.next/types` 或 `pages-manifest`，先怀疑构建缓存污染
  - 清理 `.next` 后再判定是否真是源码回归
- 关联文件：
  - `tsconfig.json`
  - `.next/`
  - `AGENTS.md`

## [007] AI 队列状态不是唯一验收信号，必须同时验字段回写与快照落库

- 时间：2026-03-21
- 场景：`user_error_diagnosis` apply 后验收
- 现象：
  - 命令行提示 `analysis:apply` 已成功
  - 但第一次快速查库时，`analysis_queue.status` 看起来还像没有更新
  - 容易误判为 apply 没生效
- 根因：
  - AI 链路的真实结果不只体现在 `analysis_queue.status`
  - 单题诊断至少还会改动：
    - `user_errors.ai*`
    - `analysis_snapshots.inputSummary`
    - 可能新增或复用 `user_strategy_refresh` 队列任务
  - 如果只盯一个状态字段，很容易在中间态或误查时得出错误结论
- 修复：
  - 本轮把 AI 验收口径固定为三步：
    1. 查 `user_errors` 是否写回诊断字段
    2. 查 `analysis_snapshots.inputSummary` 是否有审计元数据
    3. 再查 `analysis_queue.status/resultId`
- 预警信号：
  - 命令行说成功，但单看一条队列状态又觉得不对
  - 用户和开发者对“到底算没算成功”产生分歧
- 防再犯规则：
  - AI apply 验收不允许只看 `analysis_queue`
  - 对 `user_error_diagnosis` 必须同时验：
    - `user_errors.aiRootReason / aiActionRule / customAiAnalysis`
    - `analysis_snapshots.inputSummary`
    - 如为高风险，还要验是否回流到 `user_strategy_refresh`
- 关联文件：
  - `analysis-worker/src/index.mjs`
  - `docs/architecture/AI进化路线图.md`
  - `docs/progress/current_snapshot.md`

## [009] analysis:export 快捷命令如果参数协议不统一，会表现成“假导出”

- 时间：2026-03-21
- 场景：本地重导出现有 `analysis_queue` 任务 bundle
- 现象：
  - 执行 `npm run analysis:export -- --task-id=<id>` 时
  - 终端却走了默认 worker 路径，表现成 `enqueued 0 / no pending task to process`
  - 看起来像“导出失败”，其实是参数没有被导出分支正确吃到
- 根因：
  - worker 真正识别的导出参数是 `--export-task-id`
  - 但人工使用时更自然地会写 `--task-id`
  - 主流程里 `exportTaskId` 判断又早于 `applyResult`，导致别名处理稍不慎就会互相干扰
- 修复：
  - worker 现在接受 `--task-id` 作为 `exportTaskId` 的别名
  - 同时把 `apply-result` 判断提前，避免和导出别名冲突
- 预警信号：
  - 执行的是“导出命令”，日志却出现默认 `no pending task to process`
  - 目录里没有新 bundle，但也没有明确报错
- 防再犯规则：
  - 快捷命令的参数命名要与用户直觉一致，不要让“导出 task”与“apply task”写法割裂
  - 任何命令别名都必须验证一次真实 CLI 行为，而不只看代码分支
- 关联文件：
  - `analysis-worker/src/index.mjs`
  - `package.json`

## [010] 新增 AI 数据层时，必须保证“未迁库前也能优雅降级”

- 时间：2026-03-21
- 场景：新增人工反馈校正层 `ai_feedback_logs`
- 现象：
  - 我们往往会先落代码，再迁库
  - 如果 worker 或 API 直接硬查新表，在 `db push` 前就会把整条 AI 链跑挂
- 根因：
  - AI worker 是高频任务
  - 它一旦因为新表不存在直接异常，就会影响本来已经稳定的诊断 / 策略刷新主链路
- 修复：
  - 本轮为人工反馈读取加了缺表降级：
    - `42P01 / 42703` 时返回空反馈
  - 把“先不伤主链”定成新增 AI 数据层的默认规则
- 预警信号：
  - 新增一张 AI 表后，老 worker 突然开始 fatal error
  - 本来不依赖新能力的主链路也一起报错
- 防再犯规则：
  - 任何新增 AI 表或字段，都要先考虑“未迁库前如何降级”
  - 高频 worker 优先使用：
    - 缺表降级
    - 空数据兜底
    - 不让新能力阻断旧主链
- 关联文件：
  - `analysis-worker/src/index.mjs`
  - `prisma/schema.prisma`
  - 新增测试文件时，先确认它们符合当前 TS 模块解析规则
  - 递归或相互引用的工具函数，优先显式写返回类型，不要赌推断
- 关联文件：
  - `src/lib/import/ocr.test.ts`
  - `src/lib/import/ocr.ts`

## [007] Playwright 自起服务时，`NEXTAUTH_URL` 如果没对齐测试地址，会制造认证链假失败

- 时间：2026-03-19
- 场景：Playwright 冒烟测试跑生产态 `next start`
- 现象：
  - 登录页基础 smoke 可以通过
  - 但 `login -> onboarding` 或套卷前置登录会出现空白页、异常跳转、或停留在错误状态
  - 看起来像业务页面坏了，实际是测试环境口径不一致
- 根因：
  - Playwright 默认起在 `127.0.0.1:3100`
  - `.env.local` 里的 `NEXTAUTH_URL` 仍是 `http://localhost:3000`
  - NextAuth 相关回调、cookie 和绝对地址判断没有和真实测试服务保持一致
- 修复：
  - 在 `playwright.config.ts` 的 `webServer.env` 中显式设置 `NEXTAUTH_URL = baseURL`
  - 同时把用例里“第一跳必须精确落到固定路径”的脆弱断言改成更贴近真实流的校验
- 预警信号：
  - 静态页面 smoke 正常，但登录后主流程冒烟表现异常
  - 手工访问正常，Playwright 跑生产态却偶发空白或跳转异常
- 防再犯规则：
  - E2E 自起服务时，凡是依赖站点绝对地址的环境变量，都要和测试 `baseURL` 对齐
  - 不要把 `localhost:3000` 当成自动化环境里的固定真相
- 关联文件：
  - `playwright.config.ts`
  - `tests/e2e/login-onboarding.spec.ts`
  - `tests/e2e/papers-practice.spec.ts`

  - 同时 `npm run start` 也可能因为 `.next/BUILD_ID` 缺失直接失败

  - 若 `build` 再暴露源码错误，继续修源码直到构建通过

  - 启动失败排查时，不允许只看 `Ready`
  - 必须同时核对：端口监听、页面可访问、构建产物是否存在
  - 如果后台 `dev` 链不稳，先恢复一条真实可访问的 `start` 服务，再回头修脚本

  - 对资料分析的文字材料与图片材料建立“新材料重置旧材料”的规则

  - 修 `DOCX` 解析时，不能再以“抽到纯文本”作为完成标准
  - 至少要验证这四类题：
    - 题干内联小公式
    - 图形推理主图
    - 图片选项题
    - 资料分析材料切换
