# 问题复盘与防再犯清单

> 作用：把真实踩过的坑沉淀成团队共享经验，避免同类错误反复出现  
> 更新规则：只记录“真实发生过、影响过交付、值得以后避免”的问题

---

## 使用规则

以下情况必须新增或更新一条记录：

- 某个问题已经真实影响了开发、测试、交付或用户体验
- 某个问题属于“第二次出现”或“容易重复出现”
- 某个问题的修复动作本身值得变成团队规则

不需要记录的情况：

- 纯临时 typo
- 没有真实影响的探索性失败
- 已知且显然的一次性本地操作失误

每条记录尽量回答 5 件事：

1. 现象是什么
2. 真正根因是什么
3. 当时是怎么修的
4. 以后怎么更早发现
5. 以后怎么避免再犯

---

## 记录模板

复制下面模板追加：

```md
## [编号] 标题

- 时间：
- 场景：
- 现象：
- 根因：
- 修复：
- 预警信号：
- 防再犯规则：
- 关联文件：
```

## [001] 不能拿 build 通过代替页面可用

- 时间：2026-03-19
- 场景：开发服务与页面可用性验证
- 现象：
  - `npm run build` 通过
  - 终端出现 `Ready`
  - 但浏览器里仍可能看到裸 HTML、样式缺失或旧页面残留
- 根因：
  - 构建成功只能说明产物可生成，不能说明浏览器实际命中的是新服务
  - `dev:restart` / 本地残留进程 / 端口漂移会让浏览器命中旧服务
- 修复：
  - 在 `AGENTS.md` 中提升“真实页面可用”的优先级
  - 补充端口监听、真实命中进程、页面渲染状态三重验证
- 预警信号：
  - 日志显示 `Ready`
  - 但页面视觉明显不符合当前代码
  - PID、日志、端口三者不一致
- 防再犯规则：
  - 不允许再用“build 通过”直接宣布可运行
  - 至少要额外验证一次真实页面渲染结果
- 关联文件：
  - `AGENTS.md`
  - `scripts/restart-dev.sh`
  - `docs/progress/current_snapshot.md`

## [002] dev:restart 的 PID / 日志 / 真实服务可能不一致

- 时间：2026-03-19
- 场景：本地开发服务重启
- 现象：
  - `.dev-server.log` 里显示 `Local: http://localhost:3000`
  - `.dev-server.pid` 有值
  - 但真实进程可能已经退出，或 3000 上挂的是旧残留服务
- 根因：
  - 旧版本重启脚本通过 `npm run dev` 后台启动，记录到的可能是包装进程，不是长期存活的 Next 进程
  - 本地环境里旧 node 监听进程未清干净时，浏览器会命中错误服务
- 修复：
  - 重启脚本改为直接启动 `next dev`
  - 验证时同时检查端口监听与实际服务来源
- 预警信号：
  - `Ready` 后端口仍不可达
  - PID 文件中的进程不存在
  - 用户页面与当前代码完全不符
- 防再犯规则：
  - `dev:restart` 只能作为“尝试启动”，不能作为“验证通过”
  - 需要把“端口监听 + 页面实际命中”作为独立检查项
- 关联文件：
  - `scripts/restart-dev.sh`
  - `AGENTS.md`

## [003] 浏览器旧 session 里的 userId 可能和本地数据库脱节

- 时间：2026-03-20
- 场景：onboarding / settings / 任何依赖 `session.user.id` 的用户态接口
- 现象：
  - 页面已登录
  - 提交时出现“保存失败”或 `No User found` / `Record to update not found`
- 根因：
  - 本地数据库重建、数据恢复或账号重建后，浏览器 JWT 里仍保留旧 userId
  - 前端看起来已登录，但后端写入目标用户已经不存在
- 修复：
  - 在 `auth.ts` 里基于 `token.name` 做 userId 自愈回填
  - 在关键接口里把 `P2025` 这类错误转成明确提示，而不是黑盒失败
- 预警信号：
  - 登录正常，但所有用户态写接口都失败
  - 错误集中出现在 onboarding / settings / review submit 等更新接口
- 防再犯规则：
  - 本地数据库有重建或恢复动作后，优先怀疑 session 漂移
  - 涉及 `session.user.id` 的关键写接口，尽量提供明确错误文案
- 关联文件：
  - `src/lib/auth.ts`
  - `src/app/api/onboarding/route.ts`
  - 其他依赖 `session.user.id` 的用户态接口

## [004] Prisma schema 写了模型，不代表 Prisma Client 已可用

- 时间：2026-03-20
- 场景：新增 `PaperPracticeSession` 及其 API
- 现象：
  - 代码里已经在用 `prisma.paperPracticeSession`
  - 但 build 报错：`Property 'paperPracticeSession' does not exist on type PrismaClient`
- 根因：
  - schema 中 relation 未写完整，导致 `prisma generate` 实际没有成功生成对应 client 能力
  - 代码和 schema 看起来都在，但生成链是断的
- 修复：
  - 补齐 `User <-> PaperPracticeSession` 的双向 relation
  - 重新执行 `prisma generate` 与 `prisma db push`
- 预警信号：
  - Prisma API 在 TypeScript 中不存在
  - `prisma generate` 报 relation validation error
- 防再犯规则：
  - 新增 Prisma model 后，不只看 schema 文件，要立即跑：
    - `npx prisma generate`
    - `npx prisma db push`
    - `npm run build`
- 关联文件：
  - `prisma/schema.prisma`
  - `src/app/api/paper-sessions/route.ts`

## [005] `.next` 构建产物污染会制造大量假错误

- 时间：2026-03-20
- 场景：Next 构建与 TypeScript 校验
- 现象：
  - `tsc` 报大量 `.next/types/... not found`
  - `build` 报 `pages-manifest.json` 缺失
  - 错误看起来像源码问题，实际是构建目录损坏
- 根因：
  - `.next` 产物处于半生成、脏状态或历史缓存污染
  - `tsc` 又把 `.next/types` 当成 include 输入，导致假失败级联
- 修复：
  - 清理 `.next`
  - 从干净状态重新执行 `build`
  - 再复跑 `tsc`
- 预警信号：
  - 错误同时涉及大量 `.next/types/...`
  - 缺失的是生成产物，不是源码文件
- 防再犯规则：
  - 如果错误集中在 `.next/types` 或 `pages-manifest`，先怀疑构建缓存污染
  - 清理 `.next` 后再判定是否真是源码回归
- 关联文件：
  - `tsconfig.json`
  - `.next/`
  - `AGENTS.md`

## [006] 测试文件写法和工具函数签名会直接卡住全仓库 typecheck

- 时间：2026-03-20
- 场景：新增 Playwright 冒烟测试、复跑 `tsc`
- 现象：
  - `npx tsc --noEmit` 因 `src/lib/import/ocr.test.ts` 的 `.ts` 扩展名报错
  - 同时 `src/lib/import/ocr.ts` 的递归工具函数因缺少显式返回类型导致隐式 `any` 报错
- 根因：
  - 现有测试文件沿用了 Node ESM 的导入写法，但项目当前 TypeScript 配置不允许带 `.ts` 扩展名的相对导入
  - 递归函数没有显式返回类型，TypeScript 在闭包/递归场景下会放大成类型推断失败
- 修复：
  - 将测试文件导入改为不带扩展名
  - 为 `toText` 补上显式 `string` 返回类型
- 预警信号：
  - 测试文件看起来只是“新加的辅助代码”
  - 但 `tsc` 却开始报项目内其它文件错误
- 防再犯规则：
  - 新增测试文件时，先确认它们符合当前 TS 模块解析规则
  - 递归或相互引用的工具函数，优先显式写返回类型，不要赌推断
- 关联文件：
  - `src/lib/import/ocr.test.ts`
  - `src/lib/import/ocr.ts`

## [007] Playwright 自起服务时，`NEXTAUTH_URL` 如果没对齐测试地址，会制造认证链假失败

- 时间：2026-03-19
- 场景：Playwright 冒烟测试跑生产态 `next start`
- 现象：
  - 登录页基础 smoke 可以通过
  - 但 `login -> onboarding` 或套卷前置登录会出现空白页、异常跳转、或停留在错误状态
  - 看起来像业务页面坏了，实际是测试环境口径不一致
- 根因：
  - Playwright 默认起在 `127.0.0.1:3100`
  - `.env.local` 里的 `NEXTAUTH_URL` 仍是 `http://localhost:3000`
  - NextAuth 相关回调、cookie 和绝对地址判断没有和真实测试服务保持一致
- 修复：
  - 在 `playwright.config.ts` 的 `webServer.env` 中显式设置 `NEXTAUTH_URL = baseURL`
  - 同时把用例里“第一跳必须精确落到固定路径”的脆弱断言改成更贴近真实流的校验
- 预警信号：
  - 静态页面 smoke 正常，但登录后主流程冒烟表现异常
  - 手工访问正常，Playwright 跑生产态却偶发空白或跳转异常
- 防再犯规则：
  - E2E 自起服务时，凡是依赖站点绝对地址的环境变量，都要和测试 `baseURL` 对齐
  - 不要把 `localhost:3000` 当成自动化环境里的固定真相
- 关联文件：
  - `playwright.config.ts`
  - `tests/e2e/login-onboarding.spec.ts`
  - `tests/e2e/papers-practice.spec.ts`

## [008] 图片题导入不能只做 OCR 识别，必须走统一入库主链路

- 时间：2026-03-19
- 场景：截图识别录题 / 图片题导入
- 现象：
  - 图片 OCR 看起来识别成功
  - 但题目只是落到错题入口，未真正进入统一题库导入链路
  - 结果会出现去重、组卷、练习队列、结果统计口径都和文件导入不一致
- 根因：
  - 图片题入口单独走了 `/api/errors`
  - 没有复用 `/api/import/confirm` 的去重、练习队列、分析队列和正式题库写入逻辑
- 修复：
  - 把 OCR 识别抽到统一模块
  - 把截图导入改为走正式导入确认链路
  - 让图片题也落到和文件导入一致的数据模型
- 预警信号：
  - “识别成功”但结果页、题库、套卷、练习队列口径对不上
  - 图片题只能在错题页看到，不能按真题导入后的路径继续流转
- 防再犯规则：
  - 任何图片题 / OCR 入口都必须复用统一导入主链路，不能再单独旁路入库
  - 验证图片题时，不能只看 OCR 返回内容，要继续验证确认入库、去重、结果统计和后续练习链路
- 关联文件：
  - `src/lib/import/ocr.ts`
  - `src/app/api/import/screenshot/route.ts`
  - `src/app/api/import/confirm/route.ts`
  - `src/app/(app)/import/screenshot/page.tsx`

## [010] “项目启动不了”要先区分脚本假成功、构建产物缺失、和源码真报错

- 时间：2026-03-19
- 场景：本地启动 / 交付前恢复可访问服务
- 现象：
  - `npm run dev:restart` 打印成功，日志里也有 `Ready`
  - 但 3000 端口并未监听，`/login` 无法访问
  - 同时 `npm run start` 也可能因为 `.next/BUILD_ID` 缺失直接失败
  - 真正继续构建后，又可能暴露出新的源码错误，比如重复定义或类型错误
- 根因：
  - 后台重启脚本可能只记录到了“尝试启动”，并没有留下真实存活服务
  - 清理过 `.next` 之后，生产态产物如果没重建，`next start` 必然失败
  - “启动失败”常常是多层问题叠在一起，不能只看第一条报错
- 修复：
  - 先用前台 `npm run dev` 证明项目代码本身能否启动
  - 再用 `curl` / `lsof` 验证端口和页面真实可达
  - 若 `start` 缺少 `BUILD_ID`，先 `npm run build`
  - 若 `build` 再暴露源码错误，继续修源码直到构建通过
  - 在当前环境下优先用 `build + start` 恢复 3000 可访问服务
- 预警信号：
  - `dev:restart` 成功，但 `curl http://127.0.0.1:3000/login` 失败
  - `.dev-server.pid` 有值，但对应进程不存在
  - `next start` 报 `.next/BUILD_ID` 缺失
  - `tsc` 报大量 `.next/types` 缺失，且紧跟在一次失败 build 之后
- 防再犯规则：
  - 启动失败排查时，不允许只看 `Ready`
  - 必须同时核对：端口监听、页面可访问、构建产物是否存在
  - 如果后台 `dev` 链不稳，先恢复一条真实可访问的 `start` 服务，再回头修脚本
  - 以后遇到“项目起不来”，默认先分层判断：脚本 / 产物 / 源码
- 关联文件：
  - `scripts/restart-dev.sh`
  - `src/lib/daily-tasks.ts`
  - `src/app/(app)/stats/page.tsx`
  - `AGENTS.md`
- 根因：
  - 旧实现只有 `skipped`
  - 去重策略是纯后端保守逻辑，没有把“跳过 / 覆盖”变成显式产品选择
  - 结果页也缺少“覆盖更新”这类用户可见反馈
- 修复：
  - 导入页和截图导入页新增 `跳过 / 覆盖低质量旧题 / 强制覆盖`
  - 后端新增 `overwritten` 统计
  - 低质量覆盖策略按“旧题质量差且新题明显更完整”触发
- 预警信号：
  - 导入结果长期只有 `skipped`，但用户明确在做“更优识别结果覆盖旧题”的动作
  - 页面里看不到重复策略选项或结果统计没有 `覆盖更新`
- 防再犯规则：
  - 重复题处理必须是显式产品决策，不能继续默默跳过
  - 任何去重优化都要同步考虑：
    - 用户是否能选择策略
    - 结果页是否能看到跳过和覆盖的区别
    - 浏览器是否命中了新页面，而不是旧残留代码
- 关联文件：
  - `src/lib/import/duplicate-policy.ts`
  - `src/app/api/import/confirm/route.ts`
  - `src/app/(app)/import/page.tsx`
  - `src/app/(app)/import/screenshot/page.tsx`
