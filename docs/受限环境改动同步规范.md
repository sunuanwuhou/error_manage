# 受限环境改动同步规范

适用场景：

- 在公司电脑或受限机器上开发
- 不能直接推送仓库
- 不能上传源码、压缩包、仓库快照
- 只能带走纯文本说明，或允许带走 patch 文件

目标：

- 不让改动丢失
- 不把“记忆同步”当成主要交接手段
- 让原项目可以低风险、可追溯地复现改动

## 优先级顺序

按以下顺序选择同步方式：

1. `patch` 文件同步
2. 纯文本变更说明同步
3. 最后才是人工凭记忆重写

## 方案 A：允许带走 patch 文件

这是首选方案，因为保真度最高。

### 未提交 commit 时

```bash
git status
git diff --stat
git diff > my-changes.patch
```

回到原项目后：

```bash
git apply my-changes.patch
```

如果想先看是否可应用：

```bash
git apply --check my-changes.patch
```

### 已提交本地 commit 时

```bash
git format-patch -1 HEAD
```

回到原项目后：

```bash
git am 0001-*.patch
```

## 方案 B：不能带走代码，但允许带走纯文本

这是第二优先级。不要只带一句“我改过 practice/page.tsx”，要带结构化说明。

### 最低保留材料

在受限机器上至少导出以下信息：

```bash
git status > change-status.txt
git diff --stat > change-stat.txt
git diff --unified=20 > change-detail.txt
```

如果不能导出完整 diff，也至少记录：

```bash
git diff --name-only
```

### 变更说明模板

每个改动点都按这个模板写：

```text
【改动标题】
目标：
影响文件：
核心函数/接口：
旧问题：
新规则：
实现方式：
验证结果：
风险与未完成：
```

### 推荐补充信息

- 关键文件路径
- 新增函数名 / 被替换的判断逻辑
- API 入参出参是否变化
- 是否改了数据库 schema
- 是否需要补数据
- 哪些测试跑过
- 哪些地方还需要浏览器手测

## 方案 C：只能手工通知原项目

如果公司规定既不能带 patch，也不能带详细 diff，就必须输出“可复现任务单”。

任务单至少包含：

- 改动目的
- 影响范围
- 关键判断逻辑
- 页面表现变化
- 验证口径
- 不要做什么

不合格示例：

- “我改了套卷模块识别”

合格示例：

- “套卷题卡和交卷检查原本直接按每题 `type/subtype` 连续分块，导致中间 1 到 2 道误判会把同一模块切碎。现改为先生成顺序模块标签，再把被前后同一模块夹住的短误判块并回，交卷检查与总结页统一使用该规则。”

## 同步回原项目时的操作要求

无论采用哪种方式，回到原项目后都要按以下顺序处理：

1. 先看受限环境导出的文件清单
2. 再按模块逐项落代码
3. 每完成一组改动就本地验证
4. 最后补快照和复盘记录

禁止做法：

- 直接凭记忆大面积重写
- 未核对影响文件就手改
- 未验证就宣称“已经同步完成”

## 原项目落地后的最低验证

- `npm run build`
- 与本次改动直接相关的最小测试
- 至少一条主流程手工验证

如果改动涉及：

- 导入：额外验证预览、确认入库、去重
- 套卷：额外验证题卡、跳题、交卷检查、总结页
- 配置：额外验证保存成功和完成态
- 登录：额外验证登录跳转和 session

## 项目内默认要求

当处于受限环境时，任何一次有价值的改动都应至少留下：

- `git status`
- `git diff --stat`
- 可复用的文字版变更说明

如果允许导出 patch，则 patch 为默认首选交接产物。
