# 开发铁规

## 1. 唯一主模型

以下业务对象只能使用一套主模型，不允许再新增同义模型：

- 导入任务：`ImportJob`
- 题库主表：`Question`
- 练习记录：`PracticeRecord`
- 套卷会话：`PaperPracticeSession`
- 错题主表：`UserError`

### 禁止事项
- 禁止新增 `ImportTask` 替代 `ImportJob`
- 禁止新增 `Attempt` 替代 `PracticeRecord`
- 禁止新增 `WrongQuestionRecord` 替代 `UserError`
- 禁止新增其他只是在命名上重复表达相同业务的新表

### 允许例外
只有在以下两个条件同时满足时，才允许新增模型：
1. 先写迁移方案
2. 明确旧模型下线方式

---

## 2. 唯一主链

当前主链固定为：

`ImportJob -> Question -> PaperPracticeSession / PracticeRecord -> UserError`

所有新需求必须先证明自己落在这条链上。

### 需求进入开发前必须回答 4 个问题
- 入口对象是什么
- 读哪张表
- 写哪张表
- 最终沉淀到哪张表

如果这 4 个问题答不清楚，不允许开始写代码。

---

## 3. 固定改动顺序

每次开发必须按下面顺序进行：

1. `schema.prisma`
2. `db push / migrate`
3. `lib / service`
4. `route`
5. `page / component`

### 禁止事项
- 禁止页面先写、后补 schema
- 禁止 route 直接写复杂业务逻辑而不经过 service 层
- 禁止 page 直接拼复杂 Prisma 查询

---

## 4. 单一服务入口

每条主链都必须有统一服务层入口，页面和接口都只调服务层。

建议目录：

- `src/lib/import/*`
- `src/lib/questions/*`
- `src/lib/practice/*`
- `src/lib/errors/*`

### 规则
- route 可以做参数校验、鉴权、状态码处理
- service 负责业务逻辑
- page 只负责展示和交互

---

## 5. 提交门禁

任何代码在提交前，必须通过：

```bash
npm run typecheck
npm run smoke:backend
npm run smoke:frontend
```

其中：
- 本地最少必须过 `typecheck + smoke:backend`
- 合并到主分支前必须全部通过

---

## 6. 单次提交边界

一次 commit 只允许主改一段链路：

- 导入链
- 题库链
- 练习链
- 错题链

### 禁止事项
- 禁止一个 commit 同时大改导入、练习、错题三段
- 禁止“顺手修一堆”但没有边界

---

## 7. 新需求进入规则

任何新需求开始前，必须先补一段“主链映射”：

```md
需求名：
入口对象：
读取模型：
写入模型：
派生模型：
是否新增字段：
是否新增接口：
是否影响已有页面：
```

没有这段映射，不准开始改。

---

## 8. 返工判定规则

出现以下任一情况，视为“没有遵守规则”，必须先收口后继续开发：

- 同一业务出现两套模型
- 同一字段在多处语义不一致
- 页面展示结构和 route 返回结构不一致
- 一个需求导致跨两条以上主链同时返工
