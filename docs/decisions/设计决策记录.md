## ADR-001: errorROI 而非"缺口最大优先"

**决策时间**：v7.x  
**状态**：已实施

**问题**：推题逻辑用什么排序？

**选项A（放弃）**：推 mastery 最低的题（缺口最大优先）  
**选项B（采纳）**：推 errorROI 最高的题（今天最接近存量化门槛的）

**原因**：  
同样花10分钟，mastery=70%的题今天练一次就能存量化；mastery=10%的题还需要5-8次。对有限时间的用户，前者性价比高出数倍。这是资源分配问题，不是努力问题。

## ADR-002: 答错不归零（mastery最低保留10%）

**决策时间**：v7.x  
**状态**：已实施

**问题**：答错后 mastery 应该清零吗？

**原因**：  
清零会抹掉用户几周的积累，一次手误导致心理崩溃，流失率极高。保留底部记忆符合认知科学——即使答错，之前建立的记忆痕迹仍然存在。`mastery = max(current - 30, 10)`

## ADR-003: 三层数据结构（ActivityLog/Snapshot/Insight）

**决策时间**：2026-03-18  
**状态**：已实施

**问题**：AI分析结果如何存储，如何防止幻觉？

**原因**：  
- DB只存状态（mastery=75%），AI分析需要过程（为什么涨，为什么降）
- ActivityLog提供完整过程，Snapshot存分析结论，Insight存可执行建议
- 三层分离使得：原始数据不可篡改，分析结论可被用户讨论校正，系统建议可被接受/拒绝

## ADR-004: Claude 优先，MiniMax 降级

**决策时间**：2026-03-18  
**状态**：已实施

**问题**：用哪个 AI API？

**原因**：  
Claude 的上下文理解和推理质量对"行测解题思维链分析"这类任务明显更好。但 ANTHROPIC_API_KEY 是独立 API 计费，用户可能优先配置 MiniMax（已有）。因此设计为自动降级：有 Claude Key 用 Claude，没有则用 MiniMax，两者都没有则 AI 功能不可用但其他功能正常。

## ADR-005: AnalysisQueue 作为分析服务入口

**决策时间**：2026-03-18  
**状态**：已实施

**问题**：分析服务怎么知道要分析什么？

**原因**：  
不用 API 调用，不用消息队列，直接用 DB 表作为任务队列。优势：零通信成本，分析服务和主项目完全解耦，分析服务任何时候启动都能继续消费，任务状态可视化（管理员页面直接看）。这就是"共享 DB 最大的好处：零通信成本，完全解耦"的具体落地。

## ADR-006: AI 分析结果必须经过人机讨论校正

**决策时间**：2026-03-18  
**状态**：已实施

**问题**：AI 会产生幻觉，分析结果能直接应用吗？

**原因**：  
不能。AI 的分析基于统计规律，可能对特定用户不适用。设计讨论机制让用户可以确认/否定每条 finding，讨论本身写回 AnalysisSnapshot 作为下次分析的上下文，AI 不会重复被用户否定的错误结论。这形成了一个人机协作的置信度校准环。
